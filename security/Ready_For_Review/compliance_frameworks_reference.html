Compliance Frameworks Reference

CIS Controls v8.1 · NIST SP 800-53 Rev 5 (Upd 1) · SOC 2 Trust Services Criteria (2022) · 2026-04-21

Overview · Purpose · Approach · How to use this document ↑ Top

This document describes three compliance frameworks commonly applied to infrastructure environments. It is written as a pre-mapping reference — the purpose is to understand each framework's structure, authentication-relevant controls, and fit for this deployment before committing to any formal alignment.

The approach is framework-agnostic first: authentication controls are implemented and documented on their own merits. Once the control record is substantive, it can be mapped to whichever framework is required. This avoids premature constraint and keeps the underlying evidence useful regardless of which framework an auditor eventually applies.

Infrastructure in scope

The authentication surface covered by this security context spans the following services on the 3-node Proxmox hyperconverged cluster and its supporting infrastructure:

  • Linux OS — PAM, SSH, sudo
  • Proxmox VE web UI and API
  • Apache Spark / YARN ResourceManager
  • Ceph RGW (S3 API) — access key management
  • Apache Airflow web UI and API
  • JupyterHub
  • Grafana
  • Nginx reverse proxy
  • Bastion VM — SSH gateway
  • Service-to-service credentials (Spark → Ceph, Airflow → YARN, Promtail → Loki)

A CIS Controls v8 · Center for Internet Security · Implementation Groups · Authentication-relevant controls ↑ Top

CIS  The CIS Controls (currently version 8, published May 2021) are a prioritized set of 18 safeguard groups published by the Center for Internet Security. They are practical, technology-agnostic, and designed to be implemented incrementally. The framework is widely used by organizations that want actionable security posture improvements without the overhead of a full compliance program.

Implementation Groups

CIS Controls are tiered into three Implementation Groups (IGs) based on organizational size and risk profile. Most small-to-medium infrastructure deployments target IG1 as a baseline and extend into IG2 for higher-risk assets.

GroupTarget ProfileSafeguard Count
IG1Small organizations, limited IT resources, low risk tolerance56
IG2Medium organizations, dedicated IT staff, moderate risk74 (cumulative)
IG3Large organizations, security specialists, high-value targets153 (cumulative)

Authentication-relevant controls

The following CIS Controls apply directly to authentication on this infrastructure. Control numbers reference CIS Controls v8.

Control 5 — Account Management Establish and maintain an inventory of accounts; disable dormant accounts
  • 5.1Establish and maintain an inventory of all accounts (IG1). Applies to Linux OS accounts, Proxmox users, Airflow users, Grafana users, JupyterHub users, and S3 access keys.
  • 5.2Use unique passwords for all enterprise assets (IG1). Each service and each operator must have distinct credentials — no shared passwords across services.
  • 5.3Disable dormant accounts within 45 days (IG1). Applies to all services listed under 5.1.
  • 5.4Restrict administrator privileges to dedicated administrator accounts (IG1). Proxmox root, Linux root, and Ceph admin keys should not be used for routine operations.
  • 5.6Centralize account management (IG2). Relevant if an LDAP or SSO layer is added to unify Airflow, JupyterHub, and Grafana authentication.
▲ Close
Control 6 — Access Control Management Assign permissions based on need-to-know; enforce least privilege
  • 6.1Establish an access-granting process (IG1). Define who can authorize new accounts on each service and what documentation is required.
  • 6.2Establish an access-revoking process (IG1). Define the off-boarding procedure for each service (SSH key removal, LDAP disable, S3 key deletion).
  • 6.3Require MFA for all externally-exposed applications (IG1). Airflow, JupyterHub, and Grafana web UIs are exposed via Nginx and qualify.
  • 6.4Require MFA for remote network access (IG1). SSH to the bastion VM is the primary remote access vector.
  • 6.5Require MFA for administrative access (IG2). Proxmox VE web UI and any direct cluster management interfaces.
  • 6.7Centralize access control (IG2). Consolidating authentication for web services behind Nginx with a single SSO provider satisfies this for web-exposed services.
  • 6.8Define and maintain role-based access (IG3). Formal RBAC definitions for Airflow (Admin, Op, Viewer), Grafana (Admin, Editor, Viewer), Ceph (per-bucket policies).
▲ Close
Control 12 — Network Infrastructure Management Relevant sub-controls: authentication to network services and bastion access
  • 12.6Use of secure network management and communication protocols (IG2). SSH v2 only on bastion; TLS on all web-exposed services (Airflow, JupyterHub, Grafana) via Nginx.
▲ Close

Governing body and publication

CIS Controls are maintained by the Center for Internet Security (CIS) and are freely available at cisecurity.org/controls. Version 8 was published May 2021. Version 8.1 is the current release as of this writing — it refines v8 safeguards without restructuring the control families; the IG tiers and control numbers cited in this section remain valid. There is no certification associated with CIS Controls — organizations self-attest or use third-party assessors.

B NIST SP 800-53 Rev 5 · NIST · Control families · Identification & Authentication family ↑ Top

NIST  NIST SP 800-53 (currently Revision 5, published September 2020) is a comprehensive security control catalog published by the National Institute of Standards and Technology. It is the primary framework for US federal information systems (required under FISMA) and is widely referenced in regulated industries. It is more exhaustive and formally structured than CIS Controls — each control has a statement, supplemental guidance, control enhancements, and related controls.

Control families

800-53 Rev 5 organizes controls into 20 families identified by two-letter prefixes. The families most relevant to authentication are:

FamilyPrefixRelevance
Identification and AuthenticationIAPrimary — covers all authentication controls
Access ControlACAccount types, least privilege, remote access
Audit and AccountabilityAULogging of authentication events
System and Communications ProtectionSCCryptographic protocols for authentication channels
Configuration ManagementCMBaseline configuration of authentication services

IA — Identification and Authentication family

The IA family is the direct authentication scope. Key controls:

IA-2 — Identification and Authentication (Organizational Users) Core MFA requirement for organizational users

Requires that the system uniquely identifies and authenticates organizational users. Enhancements apply specific requirements:

  • IA-2(1)MFA for privileged accounts. Proxmox root, Linux root, Ceph admin.
  • IA-2(2)MFA for non-privileged accounts. Standard operator accounts on all web-exposed services.
  • IA-2(12)Accept and verify PIV credentials. Applicable only in US federal contexts; not relevant here unless required by a specific client.
▲ Close
IA-3, IA-5, IA-8 — Device, Authenticator, and Non-Org User Auth Service accounts, SSH keys, S3 credentials, external users
  • IA-3Device identification and authentication. Authenticating hosts in the cluster to each other — Spark executor nodes, Promtail agents, Airflow workers. Relevant for service-to-service trust.
  • IA-5Authenticator management. Policies governing SSH key generation, rotation, and revocation; S3 access key lifecycle; Airflow and Grafana password policies.
  • IA-5(1)Password-based authentication enhancement: enforce complexity, minimum length, history, and expiry where passwords are used.
  • IA-5(2)PKI-based authentication: validate certificates, enforce path, check revocation. Relevant if TLS client certificates are used for service-to-service auth.
  • IA-8Identification and authentication for non-organizational users. JupyterHub users who are researchers or analysts rather than operators fall here.
▲ Close
AC — Access Control family (selected) Least privilege, remote access, account management
  • AC-2Account management — establish, activate, modify, review, disable, and remove accounts. Maps to the account inventory and lifecycle process across all services.
  • AC-3Access enforcement — enforce approved authorizations for logical access to system resources. RBAC enforcement in Airflow, Grafana, Ceph bucket policies.
  • AC-6Least privilege — assign minimum access required to perform authorized tasks. Airflow service account should not have Proxmox admin access; Spark should not have full Ceph admin.
  • AC-17Remote access — establish usage restrictions and document allowed remote access methods. Bastion SSH is the sole authorized remote access path; direct cluster access should be prohibited.
▲ Close

Baseline selection

800-53 defines three baselines — Low, Moderate, and High — that specify which controls are required at each impact level. For most private infrastructure deployments not subject to FISMA, the Moderate baseline is a reasonable reference point. It includes all IA controls listed above.

Governing body and publication

Published by the National Institute of Standards and Technology (NIST), freely available at csrc.nist.gov. The current authoritative version is Rev 5, Update 1. No formal certification exists — compliance is assessed by third-party auditors (for FedRAMP) or self-declared. Revision 6 is in development as of 2026.

C SOC 2 — Trust Services Criteria · AICPA · Type I / Type II · CC6 Logical Access ↑ Top

SOC 2  SOC 2 (System and Organization Controls 2) is an audit framework published by the American Institute of Certified Public Accountants (AICPA). Unlike CIS and NIST, SOC 2 is not a set of prescriptive controls — it is a set of criteria that auditors use to assess whether an organization's controls achieve specified trust service objectives. The output is a formal audit report issued by a licensed CPA.

Trust Service Categories

CategoryCodeRequired?
SecurityCCAlways required (Common Criteria)
AvailabilityAOptional — add if uptime SLAs are relevant
Processing IntegrityPIOptional — add for data pipeline accuracy requirements
ConfidentialityCOptional — add if data confidentiality commitments are made
PrivacyPOptional — add if PII is processed

For this infrastructure, Security (CC) is the primary category. Availability and Confidentiality are relevant if the cluster supports external data customers.

CC6 — Logical and Physical Access Controls

Within the Security category, CC6 is the criterion family that maps directly to authentication. Key criteria:

CC6.1 — Logical Access Security Measures Controls to protect against unauthorized access

The entity implements logical access security software, infrastructure, and architectures over protected information assets to protect them from security events. Points of focus include:

  • CC6.1 · aIdentifies and manages the inventory of information assets. Account and credential inventory across all services.
  • CC6.1 · bRestricts logical access. Authentication required on all services; no anonymous access to Airflow, Grafana, JupyterHub, or Ceph RGW.
  • CC6.1 · cIdentifies and authenticates users. Unique accounts, no shared credentials.
  • CC6.1 · dConsiders network segmentation. Bastion-only ingress; internal services not directly internet-routable.
  • CC6.1 · eManages credentials for infrastructure and software. SSH key lifecycle, S3 key rotation, Airflow/Grafana password policy.
  • CC6.1 · fUses encryption to protect data in transit. TLS on all web interfaces; SSH for remote access.
▲ Close
CC6.2 — Prior to Issuing System Credentials Registration and authorization process for new credentials
  • CC6.2 · aRegisters and authorizes new internal and external users. Formal process for adding operators, analysts, and service accounts — documented authorization before credential issuance.
  • CC6.2 · bIssues credentials and authenticators. How SSH keys are generated and distributed; how S3 keys are created and delivered.
▲ Close
CC6.3 — Role-Based Access and Least Privilege Access removal, role changes, periodic review
  • CC6.3 · aRemoves access when no longer needed. Off-boarding procedure: SSH key removal, account disable/deletion on each service.
  • CC6.3 · bReviews appropriateness of access at defined intervals. Periodic access review schedule — who has what on each service, confirmed by an authorized reviewer.
▲ Close

Report types

TypeWhat it coversAudit period
Type IControls are suitably designed as of a specific datePoint in time
Type IIControls are suitably designed and operating effectively over a period6–12 months (typical)

Type II is what clients and partners typically request. A Type I is sometimes used as a first-year interim report while evidence accumulates for Type II.

Note: SOC 2 criteria are deliberately non-prescriptive. The same criterion can be satisfied by many different technical implementations. The audit tests whether controls achieve the criteria, not whether specific tools are in place. This makes SOC 2 well-suited to a framework-agnostic approach — document controls in plain terms first, then let the auditor map them.

Recommendation

Build the authentication control record using CIS Controls v8 as the organizational reference. The IG1 and IG2 safeguards in Controls 5 and 6 cover the full authentication surface of this deployment and produce documentation that is directly useful for operations. Once the record is substantive:

  • Map to NIST 800-53 IA / AC if a government client or regulated-industry partner requires it — the mapping is mechanical.
  • Map to SOC 2 CC6 if a client requests a SOC 2 report — engage a CPA at that point; the control record you've already built is the evidence base.
Takeaway: CIS Controls v8 is the right operational anchor. NIST 800-53 and SOC 2 are additive mappings applied later when a specific external requirement arises.
