Overview & Key Differences from Production

The development calculator sizes a 3-node Proxmox cluster with different hardware from production: single-socket (32 cores, 1 NUMA domain), 384 GB RAM, and 7× 3.84 TB SSD-class NVMe drives per node. The architecture is otherwise identical — YARN on bare Proxmox VMs, Ceph RGW (S3) for object storage, Ceph RBD for VM images.

Key difference from production: Single-socket nodes have only one NUMA memory domain. Running 2 VMs per node provides no memory locality benefit and would leave only ~1–2 executors per VM. The calculator defaults to 1 VM per node, giving each VM a large 127 GB executor slot and 5 total executors cluster-wide. The NUMA pinning toggle is retained for consistency but NUMA-related warnings are suppressed since they don't apply to single-socket hardware.

Hardware Constants (top bar)

These six values describe the physical hardware and cannot be changed in the calculator. The development-specific values to note:

  • Cores / node 32 — single 32-core CPU (vs. 64 in production). After ~14c of infrastructure overhead, only ~18c are available for VMs.
  • NUMA domains / node 1 — single socket, single memory domain (~384 GB). No benefit from NUMA-based VM splitting. NUMA warnings suppressed.
  • RAM / node 384 GB — ~40 GB reserved for infrastructure leaves ~344 GB available for the worker VM.
  • SSDs / node 3× 480 GB — ZFS mirror for Proxmox OS plus 1 hot spare. Not used for VM scratch or Ceph.
  • NVMe / node 7× 3.84 TB — split between Ceph OSDs/RBD and local scratch. Note: 7 drives vs. 8 in production; OSD slider max is 7.

Infrastructure Parameters (left card)

VMs per node (1–2, default: 1)

How many Spark worker VMs run on each physical node. With only one NUMA domain per node, splitting into 2 VMs divides the 18 available cores into ~9 each, yielding only 1 executor per VM at the default 5 cores/executor. The default of 1 VM per node is strongly recommended — it gives 2 executors per VM with a large 127 GB slot each.

vCPUs per worker VM (8–32, default: 16)

Virtual cores assigned to each worker VM. The label reads Max 32 (single socket) — since all 32 cores share one memory domain there's no NUMA penalty. After infrastructure overhead (~14c) the Effective max shown in the overhead table is 18c. The default of 16 leaves a 2-core buffer above VM allocations.

RAM per worker VM (64–360 GB, default: 256)

Memory assigned to the worker VM. Inside the VM, 2 GB is reserved for the YARN NodeManager daemon; the remainder is divided among executors. The slider maximum is 360 GB — with 384 GB total and ~40 GB reserved, up to ~344 GB is available. The default 256 GB leaves 88 GB of comfortable headroom.

NVMe drives → Ceph OSDs (4–7, default: 4)

Note the slider maximum is 7, not 8. The default of 4 OSDs yields 15.4 TB usable at rep=3 — over 15 days of ingest at 1 TB/day, far exceeding the 1–2 day dev retention requirement. The remaining 3 drives give 11.5 TB scratch per VM.

Ceph replication factor (2–3, default: 3)

3× survives 2 OSD failures (33% usable). 2× survives 1 failure (50% usable — 50% more capacity). For dev, rep=2 is acceptable since ingest data is transient and re-generation from source is easy.

NUMA pinning (toggle, default: on)

On single-socket nodes this has no performance impact. Retained for consistency with production. Both NUMA-related warnings are suppressed for dev nodes.

AQE enabled (toggle, default: on)

Enables Spark 3.x Adaptive Query Execution. When on, spark.sql.shuffle.partitions is set to a high ceiling and three spark.sql.adaptive.* params are set to true. AQE coalesces partitions down at runtime based on actual data size. Leave on unless running Spark 2.x.

YARN services VM (toggle, default: on)

Accounts for the cluster-wide YARN ResourceManager, Spark History Server, and Hive Metastore + PostgreSQL VM. Cost is amortized ÷3 across the per-node budget model. The sub-sliders set its size (default: 6 vCPUs, 16 GB RAM). Protected by Proxmox HA — if Node 1 fails, it restarts on a surviving node within ~30–60 seconds.

Spark Executor Parameters (right card)

These control how each worker VM's resources are divided into Spark executor containers. Inside every worker VM, 2 cores and 2 GB RAM are first reserved for the guest OS and YARN NodeManager. The formula: executors_per_vm = floor((vCPUs − 2) / exec_cores).

  • spark.executor.cores (2–10, default: 5) Cores per executor. With 16 vCPUs and 2 reserved: floor(14/5) = 2 executors/VM. Set to 4 for 3 executors: floor(14/4) = 3.
  • Overhead fraction (8–20%, default: 10%) JVM non-heap overhead as % of executor.memory. Maps to spark.executor.memoryOverheadFactor. Increase to 15–20% for PySpark workloads.
  • Off-heap size (0–16 GB, default: 6) Memory outside JVM heap for Parquet/Tungsten. Maps to spark.memory.offHeap.size. Subtracted before executor.memory is computed.
  • Daily data volume (0.5–4 TB, default: 1) Dev range is 0.5–4 TB (vs. 1–8 TB in production). Drives days-of-ingest, shuffle estimate, and partition count.
  • Shuffle amplification (1–4×, default: 2) Intermediate data growth from joins/aggregations. 1 TB × 2× = 2 TB shuffle must fit on scratch. With 11.5 TB scratch, this is not a concern at dev volumes.
  • Target partition size (64–512 MB, default: 200) Desired shuffle partition size. partitions = ceil(daily_TB × amp × 1,048,576 / this). With AQE on, used as a ceiling that coalesces down at runtime.

Fixed Overhead Reservations (per node)

This table shows exactly where each core and GB of RAM goes before VM sizing. It updates live as you adjust sliders. Key rows to understand:

  • Proxmox hypervisor Fixed 2c / 8 GB — host OS, QEMU process, live-migration buffers.
  • Ceph OSDs (N×) 1c + 3 GB per OSD. At default 4 OSDs: 4c / 12 GB. Updates immediately when you move the OSD slider.
  • Ceph RGW (S3 gateway) Fixed 4c / 8 GB — 4 frontend threads for the S3 API.
  • Ceph MON (monitor) Fixed 2c / 6 GB — 1 monitor daemon per node (Paxos quorum member).
  • YARN services VM Amortized ÷3. At defaults (6 vCPU / 16 GB): 2c / 6 GB per node. Row disappears when toggle is off.
  • Total reserved Sum of all above. At defaults: 14c / 40 GB.
  • Available for VMs 32c − 14c = 18c available  /  384 GB − 40 GB = 344 GB available.
  • Effective max / VM Hard ceiling for the vCPU and RAM sliders. At defaults with 1 VM/node: 18c / 344 GB. This is the true limit — not the hardware maximum of 32c.

Resource Budget (per node)

Two stacked bar charts show how the node's 32 cores and 384 GB RAM are allocated across four segments: Proxmox OS (dark grey), Ceph+RGW (amber), Spark VMs (blue), and Unallocated (green). Legend labels show live values including actual counts. If VMs claim more than available, the free segment turns red and reads OVERCOMMIT.

The three metric tiles confirm at default settings: 3 total worker VMs (1 per node), +2c free, and +88 GB free — a comfortable buffer on the constrained dev hardware.

Executor Math

Shows executors per VM, total executors cluster-wide, and total executor cores (parallelism ceiling). At default settings: 2 executors/VM × 3 VMs − 1 driver = 5 total executors, each with a 127 GB slot and 104g executor heap. The memory breakdown bar visualises how the slot is split: executor heap (blue), JVM overhead (amber), off-heap (purple). OOM risk shows LOW with a 6.6 GB safety margin.

Note: The dev cluster has far fewer executors than production (5 vs ~29) but each is much larger (104g vs ~49g). This is appropriate for development — fewer, wider executors simplify debugging and allow larger intermediate datasets without spilling to disk. To increase executor count, reduce spark.executor.cores from 5 to 4: floor(14/4) = 3 executors per VM → 8 total cluster-wide.

Storage Layout (per node)

Shows all 10 drives per node and their assigned roles. Two 480 GB SSDs form the Proxmox OS ZFS mirror; the third is a hot spare. The 7 NVMe drives are split based on the OSD slider: 4 are Ceph OSDs (labeled S3 + RBD, 3.84 TB each) and the remaining 3 become local scratch for the worker VM.

Three metrics summarise the outcome at default settings:

  • Ceph usable 15.4 TB — 12 OSDs × 3.84 TB raw ÷ rep=3. Sufficient for well over a week of 1 TB/day ingest.
  • Days of ingest 15.4 days at 1 TB/day. Only 1–2 days of retention is required for this daily ETL workload.
  • Scratch / VM 11.52 TB — 3× NVMe drives passed through to the worker VM. Worst-case shuffle at default settings: 2 TB.

The OSD chip diagram below the metrics shows the exact drive assignment across all 3 nodes, with green chips for Ceph OSDs and blue chips for scratch drives.

Spark Configuration (spark-defaults.conf)

Ready-to-paste Spark parameters in two categories. The panel heading explains which are computed vs. static:

Computed from sliders:

  • spark.masteryarn — fixed, implied by the YARN services VM architecture.
  • spark.submit.deployModecluster — driver runs inside a YARN container, not on the submitter.
  • spark.executor.coresDirect from executor cores slider (default: 5).
  • spark.executor.memoryComputed from RAM/VM, executors/VM, overhead fraction, and off-heap. Default: 104g.
  • spark.executor.memoryOverheadFactorDirect from overhead fraction slider (default: 0.10).
  • spark.memory.offHeap.enabled / sizeEnabled when off-heap slider > 0. Default: true / 6g.
  • spark.executor.instancesComputed from VMs/node, vCPUs, exec_cores, node count. Default: 5.
  • spark.default.parallelismComputed as total exec cores × 2. Default: 50.
  • spark.sql.shuffle.partitionsComputed from daily TB, shuffle amp, partition size, AQE toggle. Default: 10,923 (AQE coalesces down).
  • spark.sql.adaptive.*All three adaptive params set to true when AQE toggle is on.

Static best-practice defaults (not tied to any slider):

  • spark.sql.parquet.compression.codecsnappy — Snowflake dev instance compatibility.
  • spark.hadoop.fs.s3a.endpointPlaceholder for your dev Ceph RGW address (e.g., https://<rgw-vip>:7480).
  • spark.hadoop.fs.s3a.path.style.accesstrue — required for Ceph RGW / non-AWS S3.

Diagnostics & Warnings

At default settings the only active warning is LOW EXECUTOR DENSITY PER VM — with 2 executors per VM, a single failed executor represents 50% of that VM's capacity. This is expected on constrained dev hardware. To clear it, reduce spark.executor.cores to 4 for 3 executors/VM.

Other warnings that can appear as you adjust sliders:

  • Core overcommitVMs claim more cores than available after reservations. Reduce vCPUs/VM, reduce VMs/node, or increase OSD count (frees cores by reducing per-OSD overhead... wait, more OSDs = more overhead). Best fix: reduce vCPUs/VM.
  • RAM overcommitVMs claim more RAM than available. Reduce RAM/VM slider.
  • Executor OOM — jobs will crashexecutor.memory × (1+overhead) + off-heap exceeds slot size. Reduce off-heap or overhead fraction.
  • OOM margin very thinLess than 4 GB safety margin per slot. Fires before OOM to give early warning.
  • Ceph capacity critically low / limitedFires at <3 days (<7 days for warning). Increase OSD count, reduce rep factor, or lower daily TB.
  • Scratch disk may be insufficientEstimated shuffle data exceeds half of scratch capacity. Reduce shuffle amp or decrease OSD count to free more scratch.
  • Configuration looks healthyGreen — all checks pass. Fires when executor count ≥ 6, OOM margin ≥ 4 GB, no overcommit, and Ceph retention ≥ 7 days.
Tip: The "healthy" green indicator requires ≥ 6 total executors. On the dev cluster with only 5 executors at default settings, this will not fire even when all other conditions are met. This is by design — the threshold reflects a minimum viable production configuration, not a dev constraint.
