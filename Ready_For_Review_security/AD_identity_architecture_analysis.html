Evaluating Identity Architecture for the Spark Platform

A Reassessment of Active Directory Integration with Okta and Keycloak

Executive Summary

The Spark platform's authentication requirements present a genuine architectural tension: Kerberos-based authentication is required for runtime workloads (Spark, YARN, HDFS), while modern control-plane tooling (Airflow, Grafana, APIs) expects OpenID Connect (OIDC). Two candidate architectures resolve this differently—Active Directory with Okta (SaaS), and Active Directory with Keycloak (self-hosted). This analysis compares both on technical merit, operational complexity, and synchronization requirements. Neither solution is categorically superior; the choice depends on organizational constraints around cloud connectivity, on-premises infrastructure, and centralized identity governance.

Problem Statement

The Spark platform authentication layer must satisfy two non-negotiable requirements simultaneously:

  • Kerberos for workloads: Spark on YARN authenticates exclusively with Kerberos at runtime¹. Multi-tenancy, queue isolation, and HDFS permissions all depend on Kerberos identities.
  • OIDC for control plane: Airflow, Grafana, and service-to-service APIs require modern token-based authentication². These components neither speak Kerberos nor expect direct LDAP binds.

Both architectures acknowledge this duality. The difference lies in how identity state is managed across the two domains.

Technical Architecture Comparison
Scenario 1: Active Directory with Okta

In this architecture, Active Directory remains the Kerberos authoritative source, and Okta acts as a separate OIDC authorization server.

Authentication Flows
  • Kerberos: Spark → AD KDC (direct)³
  • OIDC: Airflow/Grafana → Okta
  • Machine-to-Machine: Services → Okta (client credentials)
Identity Synchronization
  • Active Directory is authoritative for users and groups
  • Okta syncs from AD via agent⁴
  • Sync interval is configurable (periodic or event-triggered incremental imports)
Scenario 2: Active Directory with Keycloak

In this architecture, Active Directory remains the single authoritative identity source, and Keycloak acts as an OIDC federation layer. (Note: This comparison assumes Keycloak is configured in federation mode, querying AD dynamically rather than importing users. While Keycloak can be misconfigured to cache or import users—creating a sync boundary similar to Okta—deliberate federation mode is the recommended approach for unified identity management.)

Authentication Flows
  • Kerberos: Spark → AD KDC (direct)³
  • OIDC: Airflow/Grafana → Keycloak (federated from AD)
  • Machine-to-Machine: Services → Keycloak (client credentials)
Identity Synchronization
  • Active Directory is authoritative for users and groups
  • Keycloak federates from AD via LDAP⁵
  • No sync boundary; real-time query of AD
Critical Clarification: Kerberos Runtime Authentication

In both architectures, Spark's runtime Kerberos authentication bypasses the OIDC layer entirely. Spark obtains Kerberos tickets directly from Active Directory's KDC using service keytabs³. Neither Okta nor Keycloak participates in this exchange. The difference between the two architectures lies entirely in the OIDC control plane, not in Spark's Kerberos path.

Operational Impact Analysis
Dimension AD + Okta AD + Keycloak
Identity Synchronization Okta AD Agent performs scheduled or event-triggered incremental imports⁴. Possible lag between AD changes and Okta state. Keycloak queries AD via LDAP on authentication. No sync lag; always reflects current AD state.
Group Membership Updates AD group changes → sync delay → Okta-issued OIDC tokens reflect old state until sync completes. AD group changes → immediate OIDC claim reflection (Keycloak re-queries AD on next auth event).
User Deprovisioning Disable in AD, wait for sync, explicitly disable in Okta. Two control planes, two operations. Disable in AD. Keycloak's next LDAP query returns empty result; user is immediately locked out.
Audit Trail Coherence OIDC events in Okta logs, Kerberos events in AD logs. Correlation requires manual cross-referencing of principal names. OIDC events pass through Keycloak (queryable), Kerberos in AD. One federation point with shared session identity.
External Connectivity Okta is SaaS. Requires outbound HTTPS to okta.com. Incompatible with air-gapped environments. Keycloak is self-hosted. No external SaaS dependency. Compatible with on-premises, air-gapped, or restricted networks.
Kerberos Support Okta does not provide native SPNEGO/Kerberos authentication that applications can delegate to directly⁶. Users must authenticate to Okta separately from their Kerberos session. Keycloak supports SPNEGO/GSSAPI for transparent browser SSO post–desktop login⁷.
Component Count AD + Okta = 2 identity control planes to monitor, patch, and audit. AD + Keycloak = 1 authoritative source (AD) + 1 federation layer (Keycloak) = less operational surface.
Specific Impacts for Our Infrastructure
Airflow Integration

Airflow's Flask AppBuilder auth backend supports both Okta and Keycloak natively without an intermediary⁸. Configuration complexity is equivalent for both.

Grafana Integration

Grafana provides dedicated OIDC configuration for Okta⁹ and also supports Keycloak via its Generic OAuth option⁹. Both are native, no intermediary required.

Service-to-Service Authentication

Both Okta¹⁰ and Keycloak support OAuth2 client credentials (machine-to-machine) flows. Implementation is comparable in both cases.

Synchronization Dynamics
The Sync Boundary Problem (AD + Okta)

Scenario: A user's group membership changes in AD (e.g., removal from the "analytics" group for Spark queue isolation).

In AD + Okta: The change must propagate via Okta's agent sync (periodic interval or event-triggered incremental import). Until sync completes, Okta's view of the user's groups is stale. Any OIDC token issued during this window reflects the old group membership. If the user holds a long-lived token or the sync frequency is hourly, the user may retain elevated access for an extended period.

Impact: Weak compliance posture for access revocation. Not an outage, but a delayed enforcement window.

Real-Time Reflection (AD + Keycloak)

Scenario: The same user's group membership changes in AD.

In AD + Keycloak: Keycloak's next LDAP query (on the user's next authentication event) reflects the change immediately. The next OIDC token issued includes the updated group list. Short-lived tokens mean the access denial is enforced within minutes to hours, depending on token lifetime.

Impact: Stronger compliance posture. Access control changes take effect at the next auth cycle.

Assessment and Recommendation

Both architectures are technically sound. The choice depends on our organizational constraints:

If our constraint is… Recommended architecture Rationale
On-premises infrastructure; air-gapped or restricted egress AD + Keycloak Okta requires outbound HTTPS to SaaS. Keycloak runs self-hosted with no external dependency.
Strict access revocation SLA (e.g., < 1 hour compliance window) AD + Keycloak Keycloak reflects AD changes in real-time via LDAP queries. Okta introduces sync delay.
Minimal operational surface (fewest components to patch/monitor) AD + Keycloak Keycloak is a federation layer, not a second authoritative source. Lower complexity.
SaaS identity provider preferred; minimal on-prem infrastructure AD + Okta Okta provides managed updates, enterprise support, conditional access policies, and advanced MFA.
Transparent browser SSO (users expect single sign-on without re-auth) AD + Keycloak Keycloak supports SPNEGO/GSSAPI browser SSO. Okta does not; users authenticate separately to Okta.
Conclusion

The assertion that "Kerberos is handled by the OS, so Okta is sufficient" is technically correct at the protocol level. The OS does manage Kerberos ticket acquisition and renewal. However, this does not address the architectural distinction: whether the OIDC identity control plane is unified (federated from a single source) or fragmented (synchronized from multiple sources).

For our specific infrastructure (on-premises Proxmox, separate Airflow server, Spark cluster), the key differentiator is synchronization dynamics and operational complexity:

  • AD + Okta introduces a sync boundary. Identity state in Okta may lag behind Active Directory for periods ranging from minutes to hours. Access revocation is delayed.
  • AD + Keycloak maintains a single authoritative source. Identity state is always current (within the refresh window of an LDAP query). Access revocation is immediate.
  • Both support our tooling (Airflow, Grafana, service-to-service). Integration complexity is equivalent.
  • Keycloak is infrastructure-native. Okta requires SaaS connectivity incompatible with air-gapped scenarios.

Neither architecture is categorically wrong. The choice reflects trade-offs between managed services (Okta) and operational control (Keycloak).
