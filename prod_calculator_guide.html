Saving & Resetting

Two buttons in the page header persist your slider and toggle state between sessions using localStorage — no file is written to disk.

  • Save — stores all current slider positions and toggle states. The button briefly shows Saved ✓ as confirmation. State is restored automatically the next time the page is loaded.
  • Reset — clears saved state and restores all controls to their factory defaults, then recalculates.

State is stored under the key spark_calc_prod so it does not conflict with the development calculator. Works whether the page is served via python3 -m http.server or opened directly from the filesystem.

Hardware Constants (top bar)

These six values describe the physical hardware and cannot be changed in the calculator. They represent the fixed 3-node production cluster.

  • Physical nodes: 3 — Proxmox hyperconverged cluster.
  • Cores / node: 64 — 2× 32-core CPUs. With ~16c of infrastructure overhead, ~48c are available for VMs.
  • NUMA domains / node: 2 — Each CPU socket is its own memory domain (~364 GB). Running 2 VMs/node, one pinned per domain, maximises memory locality.
  • RAM / node: 768 GB — ~42 GB reserved for infrastructure leaves ~726 GB available for VMs. Adjustable via the RAM per node slider (128–1536 GB, step: 32 GB, 24 slots × 32 GB DIMMs).
  • SSDs / node: 2× 480 GB — ZFS mirror for Proxmox OS only. Not used for VM scratch or Ceph.
  • NVMe / node: 9× 3.2 TB — Split between Ceph OSDs/RBD and local scratch. The OSD slider controls the split.

SSDs are dedicated to the Proxmox host OS. VM images are stored on Ceph RBD (backed by NVMe OSDs) to enable Proxmox HA and live migration. Scratch storage for Spark shuffle spill comes exclusively from NVMe drives not assigned to Ceph.

Infrastructure Parameters (left card)

VMs per node (1–2, default: 2)

How many Spark worker VMs run on each physical node. With NUMA pinning enabled, 2 VMs is ideal — one pinned to each NUMA domain for optimal memory locality. Setting this to 1 wastes half the NUMA bandwidth.

Impacts: Total worker VMs (×3 nodes), executor count, free cores/RAM per node, scratch disk per VM.

vCPUs per worker VM (8–32, default: 22)

Virtual cores assigned to each worker VM. Capped at 32 because that's one NUMA domain. The Effective max row in the overhead table shows the real ceiling after infrastructure reservations — typically lower than 32. The default of 22 is chosen to avoid core overcommit after accounting for the Services VM (Airflow included) overhead.

Impacts: Executors per VM (more vCPUs = more executors), core budget (may overcommit if set too high), free cores.

RAM per node (128–1536 GB, step: 32 GB, default: 768)

Sets the total physical RAM installed per node. Stepping in 32 GB increments reflects the 24-slot × 32 GB DIMM configuration. Changing this value dynamically updates the RAM per worker VM slider's maximum and proportionally rescales its current value to maintain the same relative position. All downstream calculations — available RAM, executor slot size, OOM margin — update immediately.

RAM per worker VM (dynamic max, step: 32 GB, default: ~87.5% of available)

Memory assigned to each worker VM. Inside the VM, 2 GB is reserved for the YARN NodeManager daemon, and the rest is divided among executors.

Impacts: Executor memory (more RAM = larger spark.executor.memory), OOM margin, RAM budget.

NVMe drives → Ceph OSDs (0–9 per node, default: 4)

How many of the 9 NVMe drives per node are used as Ceph storage (S3 object store + RBD for VM images). The remaining drives become local scratch for worker VMs. The slider now starts at 0, allowing a scratch-only configuration with no Ceph storage.

With the default of 4 OSDs and rep=3, usable Ceph capacity is 12.8 TB — sufficient for 6+ days of ingest at 2 TB/day. The remaining 4 NVMe drives provide 6.4 TB of scratch per VM (with 2 VMs/node).

Impacts: Ceph usable capacity and days-of-ingest (more OSDs = more S3 storage), scratch per VM (fewer OSDs = more scratch), Ceph overhead (1 core + 3 GB per OSD). This is the primary trade-off — S3/RBD capacity vs. shuffle scratch space.

Ceph replication factor (2–3, default: 3)

Number of copies Ceph stores for each object. 3× survives 2 simultaneous OSD failures (33% usable). 2× survives 1 failure (50% usable — 50% more capacity). Since ingest data is transient (flushed to Snowflake daily), rep=2 may be acceptable for the S3 pool. The RBD pool backing VM images should remain rep=3 — Ceph supports per-pool replication settings.

NUMA pinning (toggle, default: on)

When enabled, each VM is pinned to a single NUMA domain so all its memory accesses are local. Disabling it allows the hypervisor to schedule memory across both NUMA domains, adding ~2× latency on roughly half of memory accesses. Always leave this on.

AQE enabled (toggle, default: on)

Assumes Spark 3.x Adaptive Query Execution is active. AQE auto-tunes shuffle partitions at runtime, coalesces small partitions after shuffles, and rewrites skew joins — all without manual tuning.

  • On: spark.sql.shuffle.partitions is set to a high ceiling (max(2000, calculated)), trusting AQE to coalesce down. Three spark.sql.adaptive.* params are set to true.
  • Off: Partition count is the raw calculated value. Fires a warning. Leave on unless running Spark 2.x.

YARN services VM (toggle, default: on)

Accounts for a dedicated VM hosting the YARN ResourceManager, Spark History Server, Hive Metastore, PostgreSQL, and Airflow (workflow orchestrator for DAG-based ETL pipelines). Runs on one node; cost is amortized ÷3 across the per-node budget.

  • Services VM vCPUs (4–8, default: 8)
  • Services VM RAM (8–24 GB, default: 24)
Resilience note: Because VM images reside on Ceph RBD, the services VM is protected by Proxmox HA. If the hosting node fails, Proxmox automatically restarts the services VM on a surviving node within ~30–60 seconds. spark.yarn.maxAppAttempts handles in-flight job recovery. For daily batch ETL, this is sufficient without the complexity of full YARN RM HA.

Spark Executor Parameters (right card)

These control how each worker VM's resources are divided into Spark executor containers. Inside every worker VM, 2 cores and 2 GB RAM are first reserved for the guest OS and YARN NodeManager; the remainder is divided among executors.

  • spark.executor.cores (2–10, default: 5) Cores per executor. 5 is the S3 sweet spot — enough parallelism for concurrent reads without GC pressure. Formula: executors_per_vm = floor((vCPUs − 2) / this).
  • Overhead fraction (8–20%, default: 10%) JVM non-heap overhead as % of executor.memory. Maps to spark.executor.memoryOverheadFactor. Formula: executor.memory = floor((slot − offheap) / (1 + overhead) × 0.95). Increase to 15–20% for PySpark.
  • Off-heap size (0–16 GB, default: 6) Memory outside JVM heap for Parquet/Tungsten. Maps to spark.memory.offHeap.size. Subtracted before executor.memory is computed.
  • Daily data volume (1–8 TB, default: 2) Expected daily ingest into Ceph S3. Drives days-of-ingest, shuffle data estimate, scratch adequacy check, and partition count.
  • Shuffle amplification (1–4×, default: 2) Intermediate data growth from joins/aggregations. 2 TB × 2× = 4 TB of shuffle must fit on scratch disks. Increase for jobs with many wide joins.
  • Target partition size (64–512 MB, default: 200) Desired shuffle partition size. partitions = ceil(daily_TB × amp × 1,048,576 / this). With AQE on, used as a high ceiling that coalesces down at runtime. 128–256 MB is the typical sweet spot.

Fixed Overhead Reservations (per node)

This table shows exactly where each core and GB of RAM goes before VM sizing. It updates live as you adjust sliders.

  • Proxmox hypervisor Fixed 2c / 8 GB — host OS, QEMU process, live-migration buffers.
  • Ceph OSDs (N×) 1c + 3 GB per OSD. At default 4 OSDs: 4c / 12 GB. Updates live when you move the OSD slider.
  • Ceph RGW (S3 gateway) Fixed 4c / 8 GB — 4 frontend threads for the S3 API.
  • Ceph MON (monitor) Fixed 2c / 6 GB — 1 monitor daemon per node (Paxos quorum member).
  • YARN services VM Amortized ÷3. At defaults (8 vCPU / 24 GB — includes Airflow): 3c / 8 GB per node. Row disappears when toggle is off.
  • Total reserved Sum of all above. At defaults: 15c / 42 GB.
  • Available for VMs 64c − 15c = 49c available  /  768 GB − 42 GB = 726 GB available.
  • Effective max / VM Hard ceiling for the vCPU and RAM sliders. At defaults with 2 VMs/node: 24c / 343 GB. This is the true limit — not the hardware maximum of 32c.

Resource Budget (per node)

Two stacked bar charts show how the node's 64 cores and 768 GB RAM are allocated across four segments: Proxmox OS (dark grey), Ceph+RGW (amber), Spark VMs (blue), and Unallocated (green). Legend labels show live values including actual counts. If VMs claim more than available, the free segment turns red and reads OVERCOMMIT.

The three metric tiles below the bars confirm total worker VMs, free cores/node, and free RAM/node after all reservations.

Executor Math

Shows executors per VM, total executors cluster-wide, and total executor cores (the parallelism ceiling). The sub-label on Executors/VM shows the formula: usable vCPUs ÷ exec_cores (Nc reserved: OS+NM).

The memory breakdown bar visualises how the slot is split: executor heap (blue), JVM overhead (amber), off-heap (purple). Three metrics show the slot size, computed executor.memory, and OOM risk with the actual GB margin.

Slot size = (RAM_VM − 2 GB) / executors_per_vm
executor.memory = floor((slot − offheap) / (1 + overhead) × 0.95)
OOM margin = slot − (executor.memory × (1 + overhead) + offheap)

Storage Layout (per node)

Shows all drives per node and their assigned roles. Two 480 GB SSDs form the Proxmox OS ZFS mirror. The 9 NVMe drives are split based on the OSD slider: default 4 are Ceph OSDs (labeled S3 + RBD) and the remaining 5 become local scratch for worker VMs.

  • Ceph usable At 4 OSDs rep=3: 12.8 TB — 6+ days at 2 TB/day. Only 1–2 days retention required for this daily ETL workload.
  • Days of ingest Ceph usable ÷ daily TB. Updates live with the daily volume slider.
  • Scratch / VM At 4 OSDs, 2 VMs/node: 6.4 TB — 4× NVMe drives passed through. Worst-case shuffle at 2 TB × 2× = 4 TB.

The OSD chip diagram below the metrics shows the drive assignment across all 3 nodes, with green chips for Ceph OSDs and blue chips for scratch drives.

Spark Configuration (spark-defaults.conf)

Ready-to-paste Spark parameters in two categories.

Computed from sliders:

  • spark.masteryarn — fixed, implied by the YARN services VM architecture.
  • spark.submit.deployModecluster — driver runs inside a YARN container.
  • spark.executor.coresDirect from executor cores slider.
  • spark.executor.memoryComputed from RAM/VM, executors/VM, overhead fraction, and off-heap.
  • spark.executor.memoryOverheadFactorDirect from overhead fraction slider.
  • spark.memory.offHeap.enabled / sizeEnabled when off-heap slider > 0.
  • spark.executor.instancesComputed from VMs/node, vCPUs, exec_cores, node count.
  • spark.default.parallelismComputed as total exec cores × 2.
  • spark.sql.shuffle.partitionsComputed from daily TB, shuffle amp, partition size, AQE toggle.
  • spark.sql.adaptive.*All three adaptive params set to true when AQE toggle is on.

Static best-practice defaults (not tied to any slider):

  • spark.sql.parquet.compression.codecsnappy — Snowflake compatibility.
  • spark.hadoop.fs.s3a.endpointPlaceholder for your Ceph RGW address (e.g., https://<rgw-vip>:7480).
  • spark.hadoop.fs.s3a.path.style.accesstrue — required for Ceph RGW / non-AWS S3.

Diagnostics & Warnings

Context-aware alerts that fire based on current settings. The panel is positioned at the top of the page — below the hardware bar and above the parameter controls — so feedback is immediately visible as you adjust sliders. All warnings update on every slider or toggle change.

The panel shows the most relevant issues:

  • Core / RAM overcommitVMs claim more than available after reservations. Reduce vCPUs/VM, RAM/VM, or VMs/node.
  • Executor OOM — crashexecutor.memory × (1+overhead) + offheap exceeds slot. Reduce off-heap or overhead fraction.
  • OOM margin very thinLess than 4 GB safety margin. Fires before OOM as an early warning.
  • NUMA pinning disabledMemory may be scheduled across both NUMA domains, causing 2× latency on ~50% of accesses.
  • Low executor densityFewer than 3 executors per VM. A single failed executor becomes a large fraction of per-VM capacity.
  • Ceph capacity lowFires at <3 days (<7 days for warning). Increase OSD count, reduce rep factor, or lower daily TB.
  • Scratch insufficientEstimated shuffle exceeds half of scratch capacity. Reduce shuffle amp or free more NVMe from Ceph.
  • Configuration looks healthyGreen — all checks pass. Requires ≥ 6 total executors, OOM margin ≥ 4 GB, no overcommit, Ceph retention ≥ 7 days.
