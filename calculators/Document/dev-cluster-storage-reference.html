Storage Architecture Reference

OSD Sizing · Configuration Matrix · NVMe Drive Allocation · HPE DL385 Gen11 · 2026-04-17

Overview & Assumptions · Document scope · Data flow · Operational constraints & cadence ↑ Top

This document presents the storage architecture analysis for the HPE DL385 Gen11 development cluster (3 nodes · 32c / 384 GB / 7× 3.84 TB NVMe per node) running Apache Spark via YARN with Ceph hyperconverged storage. It consolidates two related analyses into a single reference, structured so that each section builds directly on the one before it.

Sections should be reviewed in order A through F on first reading. Section B is the pivotal section — it establishes the storage constraint that eliminates half the drive configurations from consideration and governs all conclusions in Sections C through F. Sections C and D size the two storage consumers independently (Ceph capacity and Spark scratch), and Section E applies both constraints simultaneously across the full configuration matrix. Section F states the recommendation.

The analysis reflects a specific architectural decision: CSV ingest flat files will be staged to local Ceph RGW (S3) rather than a remote cluster, as the remote option will not be available within the target deployment window. This single change is what drives the revised OSD minimum and the elimination of configurations A–E.

Data Flow

The following sequence describes the full ingest-to-Snowflake pipeline as implemented in this deployment:

  1. Raw CSV flat files are received and staged to local Ceph RGW (S3) — s3a://ingest/csv/
  2. Spark (via YARN) reads CSV partitions from Ceph RGW into executor memory across all 3 nodes
  3. Each executor task converts its partition and writes Parquet part files directly to local Ceph RGW — s3a://output/parquet/part-XXXXX.snappy.parquet. There is no intermediate staging to scratch; Parquet lands on Ceph as tasks complete
  4. When all tasks finish, Spark writes a _SUCCESS marker. The Parquet output is the collection of part files — there is no post-job assembly step
  5. A scheduled outbound batch job (initiated from within the cluster network) reads Parquet from local Ceph and transfers it to a Snowflake-accessible cloud staging location
  6. Snowflake executes COPY INTO from cloud staging — this is Snowflake-internal and requires no inbound access to the cluster network
  7. The batch job receives transfer confirmation and triggers deletion of Parquet files from local Ceph
  8. CSV files are moved off local Ceph to a storage node as an operational gate before the next ingest job begins (see Operational Constraints below)

Operational Constraints & Cadence Assumptions

The constraints below are assumed procedural requirements governing job cadence. They are not enforced by software at this time and must be observed operationally. Violation of the sequencing gate (constraint 2) risks Ceph capacity exhaustion at 5 TB ingest.
  1. Snowflake has no inbound network access. All data movement is outbound, initiated from within the cluster. The cluster does not expose any endpoint to Snowflake.
  2. New ingest jobs are gated on CSV removal. A new ingest job cannot start until the previous job’s CSV files have been moved off local Ceph to a designated storage node. This is a hard operational requirement to prevent Ceph capacity exhaustion during concurrent job overlap.
  3. Parquet retention is batch-transfer-triggered. Parquet files persist on local Ceph until the outbound batch job confirms successful delivery to cloud staging. Deletion is not Snowflake-triggered and is not automatic on job completion.
  4. Up to 2 jobs’ worth of Parquet output may coexist on local Ceph. If a new ingest job completes before the prior job’s batch transfer is confirmed, both Parquet sets are present simultaneously. The Ceph capacity sizing in Section C is based on this worst-case assumption.

Pivotal section. The architectural decision documented here drives the OSD minimum established in Section C and the elimination of Configurations A–E in Section E. All downstream conclusions depend on the constraint quantified below.

The Decision

An earlier analysis assumed raw CSV ingest files would be stored on a remote Ceph cluster, with only Parquet output landing on local Ceph. That assumption has been superseded. The remote cluster will not be available within the target deployment window for the development cluster.

CSV ingest files will be staged to local Ceph RGW (S3). This means at peak, multiple large datasets coexist on local Ceph simultaneously. CSV and Parquet have different retention triggers: CSV is removed to a storage node before the next ingest job starts; Parquet persists until an outbound batch transfer job confirms successful delivery to Snowflake cloud staging. Up to 2 jobs’ worth of Parquet output may be present concurrently. See the Overview & Assumptions section for the full data flow and operational constraints governing this cadence.

G Scratch Space: Adequacy Concern & Expansion Analysis · Config F ceiling · Scratch ceiling analysis · Compression & AQE mitigations ↑ Top
Config F (JBOD/XFS) resolves the ceiling concern that would have existed under Config I. With 19.2 TB scratch per node, Config F’s shuffle amplification ceiling at 5 TB ingest is 5.76× — the same headroom that would have required adding a drive per node under HW RAID-5. Two chained sort-merge joins on even data (typically 4×–5×) do not require developer intervention. This section documents the ceiling analysis and the mitigations that extend the practical safe envelope further.
WHY 4× IS NOT A SAFE CEILING Query patterns that push shuffle amplification above 4×

Shuffle amplification is a function of query structure, not ingest volume. A Spark job performing a single filtered aggregation over well-bucketed tables will see 1×–2× amplification. A job performing multiple chained sort-merge joins over unbucketed, skewed tables will see 5×–7× or higher. Both are normal on a development cluster.

The following query patterns drive shuffle amplification above 4× without being exceptional or poorly-written:

Query PatternTypical Shuffle AmpWhy
Single sort-merge join, even data3×–4×Both sides repartitioned; intermediate keys materialized on scratch
Two chained sort-merge joins4×–6×Each join stage re-shuffles; intermediate result also materializes
GroupBy + sort on large unbucketed table3×–5×Hash aggregate shuffle + sort-merge pass
Window function with large partition key4×–6×Full repartition by partition key; frame materialized per executor
Skewed join without broadcast hint5×–8×Skewed partitions spill repeatedly; AQE may insert additional repartition

CONFIG F CEILING ANALYSIS Maximum shuffle amplification Config F (JBOD/XFS) can sustain

Config F provides 19.2 TB scratch per node (5 drives, JBOD/XFS). The per-node scratch requirement is ingest_TB × shuffle_amp × 2 ÷ 3 nodes. Solving for the maximum shuffle amplification Config F can sustain:

max_shuffle_amp = scratch_per_node ÷ (ingest_TB × 2 ÷ 3) = 19.2 ÷ (5 × 0.667) = 5.76×

Any job running at 5 TB ingest with a shuffle amplification above 5.76× exhausts per-node scratch. Spark begins spilling to disk beyond local scratch — which on this hardware means spilling to Ceph (the only other storage path), introducing network I/O into what should be local operations and severely degrading job performance. If Ceph is also under pressure, the job fails.

OPERATIONAL DEPENDENCY What Config I requires from the development team

Without additional drives, Config I requires the following from the development team:

  • Developers must check scratch adequacy in the dev calculator before submitting any job expected to involve multiple joins or window functions at full ingest volume
  • Jobs must include broadcast hints for all lookup tables that can fit in executor memory, to avoid unnecessary sort-merge shuffles
  • Shuffle amplification must be actively monitored in the Spark UI; jobs trending above 4× should be checkpointed or rewritten before completing at full scale
  • AQE (Adaptive Query Execution) must remain enabled to catch and repartition late-stage skew before it cascades into scratch exhaustion

These are not unreasonable requirements for a mature team. They are unreasonable to treat as reliable constraints on a development cluster, where the point of the environment is to run queries that have not yet been optimised. The operational dependency is a latent risk, not a mitigated one.

COMPRESSION HEADROOM How default shuffle compression affects the effective ceiling

The scratch ceiling formula uses uncompressed daily ingest volume as its input — it is a conservative worst-case model. In practice, spark.shuffle.compress = true (default, snappy codec) compresses map output before writing to scratch, typically achieving 2–3× compression on mixed ETL data. [2] spark.shuffle.spill.compress = true (default) applies the same compression to reduce-side spill. [2]

The practical effect: actual on-disk scratch usage is approximately 40–60% of the calculated ceiling value when default compression is active. At 5 TB/day, Config F’s effective ceiling under compression is closer to 11× than the formula’s 5.76×.

Additionally, increasing spark.io.compression.lz4.blockSize from the default 32 KB to 512 KB yields up to a 20% further reduction in shuffle and spill file size, additive on top of the snappy baseline ratio. [6] This is a zero-cost configuration change applied in spark-defaults.conf.

Important: These compression gains are workload-dependent. Parquet data ingested from upstream is already compressed; shuffle data derived from it may compress less than raw CSV intermediates. [8] Safe planning sizes scratch against the uncompressed calculation and treats compression as safety margin — not a provisioning assumption. The formula is intentionally conservative.

AQE RUNTIME MITIGATION How Adaptive Query Execution reduces wide-join scratch pressure at runtime

Adaptive Query Execution (spark.sql.adaptive.enabled = true, default on Spark 3.2+) [5] re-optimizes query plans at runtime using actual statistics collected during execution. Two behaviors directly affect scratch pressure:

  • Auto-broadcast conversion: AQE monitors actual byte counts during execution. If either side of a join falls below the adaptive broadcast threshold at runtime, AQE converts a planned sort-merge join to a broadcast hash join — retroactively eliminating the shuffle for that stage. Jobs that appear as “wide shuffle joins” at plan time may execute with near-zero scratch consumption on those stages.
  • Skew join splitting: When a shuffle partition is more than 5× the median size, AQE automatically splits it into smaller sub-tasks. This prevents a single hot key from accumulating disproportionate scratch on one node and causing localized exhaustion even when the cluster average is within limits.
Caveat: AQE auto-broadcast requires accurate statistics and only applies when one join side is genuinely small. It is a runtime safety net, not a provisioning substitute. Do not rely on AQE behavior when sizing scratch capacity. The formula and ceiling values in this document remain the correct planning basis.

BUCKETING AS STRUCTURAL RELIEF Eliminating shuffle entirely on hot join paths — no hardware required

Bucketed tables are pre-sorted and pre-partitioned on a join key at write time. When two tables with the same bucket count are joined on their bucket key, Spark eliminates the shuffle Exchange node from the query plan entirely — the join executes locally on each executor with zero shuffle data written to scratch. [4]

For ETL pipelines that join the same tables on the same keys repeatedly (e.g., customer_id, account_id, product_id), writing intermediate tables as bucketed Hive-compatible tables eliminates the shuffle cost for every subsequent run after the initial write:

df_orders.write   .bucketBy(64, "customer_id")   .sortBy("customer_id")   .saveAsTable("etl.orders_bucketed")
Scratch impact: Shuffle for that join stage drops to zero. A pipeline with two or three hot join keys, once bucketed, can eliminate the majority of its scratch consumption. This is the structural answer to the Config I constraint at 5 TB/day — available without any hardware change, through application-layer schema design.

This technique requires ETL developers to be aware of the pattern and apply it deliberately. It should be covered in the developer briefing alongside broadcast hints and reduceByKey practices.

CONCLUSION Hardware procurement decision vs. operational dependency
Config I is adequate on paper but operationally fragile at 5 TB ingest. Its 15% margin at the modeled worst-case (5 TB @ 4×) is consumed by a single two-stage sort-merge join running above 4.61×. On a development cluster, that is a normal occurrence, not an exceptional one. The configuration requires the development team to actively avoid a class of queries that the cluster should be able to absorb without intervention.

Adding one 3.84 TB NVMe drive per node (3 drives, cluster-wide) and allocating it to scratch removes this operational dependency for all realistic development workloads. The ceiling rises to 5.76×, covering complex multi-join queries on even data without monitoring requirements. Only queries involving severe data skew across multiple join stages remain constrained — and those queries should be fixed at the data layer regardless of cluster sizing.

This is a hardware procurement decision, not a configuration decision. It should be evaluated against the cost of developer time lost to scratch-induced job failures and the operational overhead of enforcing per-job shuffle amplification checks on a development team.

Vendor-blessed mkfs.xfs / mount options for the four NVMe scratch drives

Ksolves' configuration document § 3.3 specifies the literal commands the vendor will run during cluster provisioning to format and mount the four scratch NVMes (Config F) on each worker VM. Captured here so fqdn can verify post-install and reproduce in case of node rebuild.

StepCommand / option
Format (per drive, drives 3–6)mkfs.xfs -f -d su=256k,sw=1 /dev/nvme[3-6]n1

Mount points/mnt/nvme1 … /mnt/nvme4 (used by yarn.nodemanager.local-dirs for shuffle/spill)

The su=256k,sw=1 stripe geometry matches single-drive JBOD; noatime,nodiratime remove access-time writes that would otherwise compete with shuffle IO; discard issues TRIM continuously to keep NVMe wear-leveling healthy; allocsize=64m reduces fragmentation under bursty Spark spill writes.
