Dev cluster · 3-node HPE DL385 Gen11 · 2026-04-19  ·  Rohn Wood
Overview Why Shuffle Matters on This Cluster · The scratch constraint in plain terms · What can go wrong ↑ Top

Shuffle is the phase of a Spark job where data is repartitioned across nodes — groupBy, join, and sort all trigger it. Shuffle data lands on NVMe scratch drives and stays there until every downstream task that reads it has finished. Unlike memory pressure, which degrades performance gradually, scratch exhaustion terminates the job.

On this cluster, scratch is the binding constraint. Each node has 7 NVMe drives; 2 are dedicated to Ceph OSDs, leaving 5 drives × 3.84 TB = 19.2 TB usable scratch per node (Config F, JBOD/XFS — no hardware RAID). The sizing model accounts for map-side shuffle output and reduce-side spill existing on disk simultaneously — a 2× factor on the raw shuffle volume. At 5 TB/day ingest, this yields a shuffle amplification ceiling of 5.76× before scratch is exhausted.

The concrete risk
A developer who writes a groupByKey on a 5 TB dataset, without map-side aggregation, can easily produce 5–7× shuffle amplification. At 5 TB/day that exceeds the cluster ceiling. The job fails mid-run, scratch fills across all three nodes simultaneously, and other concurrent jobs on the cluster are affected. This is not hypothetical — it is the default outcome of poorly written aggregation jobs on datasets of this size.

The sections below explain the mechanics, show where the budget goes, and give you the specific operator choices and configuration settings that keep amplification within bounds. Default shuffle compression (snappy) provides roughly 2× headroom beyond the calculated ceiling, but that buffer should not be part of your planning — it is not guaranteed for pre-compressed input data, and you cannot observe it without measuring your specific workload.

A How Shuffle Works · Map-side write · Reduce-side fetch and spill · Why both hit scratch at once ↑ Top

Shuffle is a two-phase operation. Understanding both phases matters because both can produce data on scratch simultaneously — which is where the 2× factor in the sizing model comes from.[3]

Phase 1 — Map-side write

During the map phase, each task computes a partial result and sorts it by partition key. The sorted output is written to a shuffle file on local NVMe scratch. This file is not cleaned up when the map task finishes — it stays on disk until every reducer that needs data from that map task has successfully read it. In a large job with many stages, that can be a long time. Map-side shuffle files accumulate on scratch across the duration of the shuffle stage.

Phase 2 — Reduce-side fetch and spill

Reduce tasks pull their partition's data from the shuffle files written by all map tasks across all nodes (via the External Shuffle Service). The fetched data is buffered in executor memory. When the buffer fills — which happens whenever the partition being reduced is large relative to the executor's memory allocation — Spark spills the in-memory buffer to a temporary spill file on the same local NVMe scratch.

Why 2× is the planning factor
The worst case is when map-side shuffle files from phase 1 are still on disk (reducers have not finished reading them) at the same time that reduce-side spill files from phase 2 are being written. Both sets of files occupy scratch simultaneously. Davidson & Or document this explicitly as the correct two-phase disk model for worst-case sizing.[3] The scratch budget formula uses it accordingly: scratch_needed = daily_TB × shuffle_amp × 2 / nodes.

Data flow

Input partitions (HDFS / Ceph S3) | v [ Map tasks ] — partial computation + sort by partition key | | shuffle write → NVMe scratch (persists until all reducers finish reading) v Shuffle files on local NVMe ←─── Phase 1 scratch usage | v (External Shuffle Service reads, sends over network) [ Reduce tasks ] — fetch into executor memory buffer | | if memory buffer fills → spill to NVMe scratch ←── Phase 2 scratch usage | v [ Output ] (written to Ceph S3 / next stage) Phases 1 and 2 scratch coexist on disk during the reduce phase. Both are compressed (snappy by default), both count against your budget.

NVMe and this cluster's hardware position

The original LinkedIn Magnet system was built because HDD-based shuffle at LinkedIn consumed ~15% of total Spark compute due to fetch latency — HDD delivers ~150 IOPS at 5 ms latency.[2] Our NVMe drives operate at 500K–1M IOPS at ~100 µs latency. Standard YARN External Shuffle Service on this hardware already operates far beyond the performance regime that motivated Magnet. The small-block IOPS problem is not applicable here. Scratch exhaustion — not IOPS — is the operational risk.

B The Scratch Budget · Config F ceiling · Amplification table · Compression buffer vs. planning budget ↑ Top

Config F scratch allocation

7 NVMe drives per node. 2 are Ceph OSDs. The remaining 5 drives are JBOD (individual pass-through devices), each formatted XFS — no hardware RAID, yielding:

Scratch per node  = 5 drives × 3.84 TB  = 19.2 TB
Scratch formula:    scratch_needed_per_node = daily_TB × shuffle_amp × 2 / nodes
Ceiling:            shuffle_amp_max         = (scratch_per_node × nodes) / (daily_TB × 2)
                                            = (19.2 × 3) / (daily_TB × 2)

At 2 TB/day: ceiling = 57.6 / 4 = 14.4×
At 5 TB/day: ceiling = 57.6 / 10 = 5.76×

The ceiling drops sharply as ingest volume increases. At 5 TB/day there is less than one stop of margin between "poorly-written job" (4–5×) and "scratch exhausted" (5.76×). A drive failure under JBOD terminates any job using that drive; resubmit once the drive is replaced and reformatted XFS.

The compression buffer

Default shuffle compression (spark.shuffle.compress = true, snappy codec) typically achieves 2–3× compression on mixed ETL data.[4] Both shuffle files and reduce-side spill files are compressed by default.[4][8] In practice, the actual on-disk usage is roughly 40–60% of the uncompressed value — meaning the effective ceiling is approximately 2× higher than the numbers in the table above when compression is working as expected.

Do not design to the compression buffer
Compression ratios are workload-dependent. Jobs reading already-compressed Parquet files may see little or no additional compression benefit on shuffle output. Jobs with high-cardinality keys or random data compress poorly. The displayed ceiling is the correct planning number. The compression buffer is a safety margin that exists, not a budget you can spend.

The LZ4 block size setting (spark.io.compression.lz4.blockSize = 524288) provides an additional concrete reduction: Pandey showed empirically that increasing the block size from the 32 KB default to 512 KB produces up to 20% smaller shuffle and spill files.[6] This is additive on top of baseline compression and is recommended in spark-defaults.conf. See Section D for the full parameter table.

C Operator Choices That Drive Amplification · reduceByKey vs groupByKey · Broadcast joins · AQE · Bucketing · repartition vs coalesce ↑ Top

Shuffle amplification is not primarily a configuration problem — it is a code problem. The same query intent, written two different ways, can differ by 3–5× in scratch consumption. The sub-sections below cover the five choices that matter most on this cluster.

C1 — reduceByKey vs. groupByKey The single highest-amplification mistake in Spark ETL

groupByKey() ships all values for every key to the reducer before doing any aggregation. If you have 50 million rows keyed on 100,000 distinct values, all 50 million rows cross the network. Spark has to buffer the complete value list for each key in memory or spill it to scratch before it can reduce. This typically produces 5–7× shuffle amplification; the same logic rewritten with reduceByKey() may fall to 2–3×.[4][6]

reduceByKey() runs a partial aggregation (combiner) on each map task before the data is written to the shuffle file. The map task computes a partial sum, max, count, or whatever the function specifies, and only the partial results cross the network. At the 5 TB/day ingest target, this single operator choice determines whether your job runs successfully or exhausts scratch.

Do not use groupByKey for aggregations
groupByKey is appropriate only when you genuinely need the complete list of values for a key (e.g., collecting all events per user into a list). If the goal is a sum, count, max, min, or any associative aggregate, use reduceByKey or aggregateByKey. 
# Bad: ships all values to reducer before summing
rdd.groupByKey().mapValues(sum)

# Good: partial sum on map side, only partial results shuffled
rdd.reduceByKey(lambda a, b: a + b)

# Also good for more complex aggregations
rdd.aggregateByKey(
    zeroValue=0,
    seqFunc=lambda acc, v: acc + v,
    combFunc=lambda a, b: a + b
)

The DataFrame API equivalent: always prefer .groupBy().agg(sum(...)) over collecting rows and aggregating in Python/Scala. Spark's Catalyst optimizer will push the aggregation down to the map side automatically for standard aggregate functions.

C2 — Sort-merge join vs. broadcast hash join Wide joins shuffle both sides; broadcast joins shuffle neither

A sort-merge join shuffles both input datasets by join key so that matching keys land on the same node. If both sides are large, this doubles the data movement and adds the full 25% sort-merge overhead on top. For a 5 TB fact table joined to a 1 TB dimension table, that is 6 TB of data shuffled across the network and written to scratch.

A broadcast hash join sends the smaller table to every executor as an in-memory hash map. The larger table is never shuffled — each partition of the large table is probed locally. Scratch impact on the join stage: near zero.

Spark's threshold for automatic broadcast is controlled by:

spark.sql.autoBroadcastJoinThreshold = 10485760  # 10 MB (default)

If a table is under 10 MB after statistics are gathered, Spark will broadcast it automatically. You can override this per query:

# Hint to the optimizer — broadcast this side of the join
from pyspark.sql.functions import broadcast
df_large.join(broadcast(df_small), "join_key")

You can also raise the global threshold. On this cluster with 384 GB RAM per node and executor memory allocated accordingly, broadcasting tables up to several hundred MB is often reasonable — but measure the memory cost before raising it cluster-wide, as every executor receives the full broadcast copy.

C3 — AQE auto-broadcast at runtime Spark 3.2+ can convert wide joins to broadcast joins mid-job

Adaptive Query Execution (AQE), enabled by default since Spark 3.2, collects actual partition size statistics after each shuffle stage completes.[5] If a join side turns out to be smaller than spark.sql.adaptive.autoBroadcastJoinThreshold at runtime — even if the query planner estimated it as large — Spark converts the sort-merge join to a broadcast hash join dynamically, avoiding the full shuffle for that side.

AQE also handles data skew: spark.sql.adaptive.skewJoin.enabled = true (default) splits partitions that are more than 5× the median partition size, reducing worst-case reduce-side spill for uneven key distributions.[5]

AQE is your safety net, not your budget
AQE can shift runtime behavior toward the lower end of the scratch range when it fires. It is not guaranteed to fire for all join patterns and cannot be relied on for provisioning. Keep AQE enabled, verify it is on (spark.sql.adaptive.enabled = true), and let it help — but plan your scratch budget assuming sort-merge joins fully materialize.

To check whether AQE fired on a specific join, look at the Spark UI's SQL/DataFrame tab for the query plan. A join annotated BroadcastHashJoin instead of SortMergeJoin indicates AQE converted it at runtime.

C4 — Bucketing Eliminates the shuffle Exchange node entirely on matching joins

Bucketing pre-shuffles and pre-sorts data at write time, persisting the result as a table.[4][5] When two bucketed tables with the same bucket count and the same bucket key are joined, Spark knows the data is already co-located by join key and skips the shuffle Exchange stage entirely. Scratch impact on that join: zero.

This is the highest-leverage optimization for any join that runs repeatedly across multiple jobs — account_id, customer_id, order_id, or any other hot join key in your schema.

# Write a bucketed table — do this once, use it many times
df.write \
  .bucketBy(64, "account_id") \
  .sortBy("account_id") \
  .saveAsTable("db.accounts_bucketed")

# Any join on account_id between two bucketed tables with the same
# bucket count will produce no shuffle Exchange node in the plan
spark.table("db.fact_events") \
  .join(spark.table("db.accounts_bucketed"), "account_id") \
  .groupBy("region") \
  .agg({"revenue": "sum"})

A few notes on making bucketing work correctly:

  • Both tables must be bucketed on the same key and the same number of buckets. A 64-bucket table joined to a 128-bucket table still shuffles.
  • Bucketing only works with Hive-compatible table storage (Parquet, ORC). It does not apply to DataFrames read directly from files.
  • Bucket count should be a multiple of the expected number of executor cores to distribute work evenly. 64 is a reasonable starting point for this cluster's core count.
  • The write is more expensive than a plain write — you pay once at write time to avoid shuffle costs across all future reads. For frequently re-joined tables, this is usually worthwhile within the first 2–3 job runs.

C5 — repartition() vs. coalesce() One triggers a full shuffle; the other does not

repartition(N) triggers a full shuffle — all data is redistributed across N partitions by hash or round-robin. This is appropriate when you need to increase the number of partitions, change the partition key, or produce evenly balanced output partitions before a wide operation.

coalesce(N) is a narrow transformation — it merges existing partitions without a shuffle by combining them on the same executor. It can only reduce the partition count, not increase it. For the common case of writing output with fewer partitions than the processing stage used, coalesce is correct and avoids the shuffle cost entirely.

# Use coalesce when you are reducing partition count for output
result.coalesce(8).write.parquet("s3a://bucket/output/")

# Use repartition when you need to increase partitions or change the key
large_df.repartition(200, "account_id").write.parquet("s3a://bucket/output/")

One caution: coalesce can produce skewed output if the source partitions are uneven in size. If you see a single large output file alongside many small ones, consider repartition instead to rebalance, accepting the shuffle cost.

D Configuration Reference · 7 recommended spark-defaults.conf settings · What each one does and why ↑ Top

These settings belong in spark-defaults.conf and apply to every job on the cluster unless overridden at the session level. Each entry includes what it does, what the upstream default is, what we set it to, and why it matters for this hardware.

Parameter Default Set to Purpose and rationale
spark.serializer Java serializer KryoSerializer Spark serializes objects when writing shuffle data and when broadcasting variables. Java serialization is general-purpose and verbose — Kryo is roughly 10× more compact for typical data types. Smaller serialized objects mean smaller shuffle files on scratch and faster network transfer. For most ETL workloads, this is the single configuration change with the largest direct impact on shuffle file size.[4][6][7]

Set: spark.serializer = org.apache.spark.serializer.KryoSerializer
spark.io.compression.lz4.blockSize 32768 (32 KB) 524288 (512 KB) Shuffle files and spill files are compressed in blocks. The default 32 KB block size was tuned for memory-constrained environments and is too small for large datasets on modern hardware — the compression algorithm cannot build an efficient model with such small windows. Increasing to 512 KB produces up to 20% smaller shuffle and spill files empirically.[6] The cost is slightly higher memory use during compression, which is negligible given this cluster's 384 GB RAM per node.

This applies to both shuffle writes (Phase 1) and reduce-side spill (Phase 2), so the savings compounds.
spark.shuffle.file.buffer 32768 (32 KB) 1048576 (1m) Each map task buffers shuffle output in memory before flushing to the NVMe shuffle file. The default 32 KB buffer causes excessive small write flushes — each flush is a separate system call and disk write. On NVMe, this matters less for latency than on HDD, but it still creates unnecessary IOPS fragmentation and CPU overhead. A 1 MB write buffer aligns better with NVMe optimal write unit sizes and reduces flush frequency substantially.[4][6] Multiple sources confirm 1 MB is the right value on clusters with adequate RAM; earlier guidance of 64–128 KB was based on more memory-constrained environments.
spark.shuffle.memoryFraction 0.20 (20%) 0.30 (30%) The fraction of executor memory allocated to shuffle read buffers (the reduce-side fetch buffer). When this buffer fills, Spark spills to scratch. The default 20% is conservative for the memory environment we have. Raising to 30% gives each reducer more in-memory room before spilling, directly reducing Phase 2 scratch usage for join-heavy and aggregation-heavy jobs.[4]

Do not raise this too far — the remaining memory fraction is used for task execution, caching, and the executor overhead allocation. On this cluster's executor sizing (typically 20–40 GB per executor), 30% is a reasonable balance.
spark.reducer.maxSizeInFlight 50331648 (48 MB) 100663296 (96m) The maximum total size of shuffle blocks fetched per reducer task per request.[4][7] The default 48 MB was sized for clusters with limited RAM. At 96 MB, each reducer makes fewer round-trip fetch requests to the External Shuffle Service, reducing fetch latency and network overhead. This is a performance setting — it does not change the total amount of data shuffled or the total scratch used — but it reduces the time reducers spend waiting for data to arrive, improving throughput per stage.
spark.sql.adaptive.enabled true (Spark 3.2+) true (verify explicitly) AQE is default-on in Spark 3.2+ but can be overridden by job-level configuration or by application code. Setting it explicitly in spark-defaults.conf ensures it stays on regardless of what individual jobs set.[5] AQE provides two scratch-relevant benefits: runtime sort-merge to broadcast join conversion (reduces shuffle volume when one side is smaller than expected), and dynamic partition coalescing (reduces the number of small shuffle partitions that cause fragmented small writes).

It does not replace provisioning — but it provides a meaningful runtime safety net.
spark.sql.adaptive.skewJoin.enabled true (Spark 3.2+) true (verify explicitly) When AQE is enabled, skew join handling splits partitions that are more than 5× the median partition size into smaller sub-partitions, distributing the work across more tasks.[5] Without this, a skewed join key can cause one or two reducer tasks to receive a disproportionate amount of data, filling their memory buffer and spilling aggressively to scratch — even when the overall job is within the amplification budget. Skew handling prevents that tail scenario from exhausting scratch on a per-partition basis. As with adaptive.enabled, setting this explicitly prevents accidental override.

Recommended spark-defaults.conf block

spark.serializer                            org.apache.spark.serializer.KryoSerializer
spark.io.compression.lz4.blockSize          524288
spark.shuffle.file.buffer                   1m
spark.shuffle.memoryFraction                0.30
spark.reducer.maxSizeInFlight               96m
spark.sql.adaptive.enabled                  true
spark.sql.adaptive.skewJoin.enabled         true

E Job Patterns: Low vs. High Scratch · Quick reference table · Common patterns mapped to scratch impact ↑ Top

Use this as a quick self-check before submitting a job at the 5 TB/day scale. Any HIGH pattern at that volume can exhaust scratch without compression luck.

Pattern Scratch impact Notes / recommended alternative
groupByKey on large dataset HIGH Ships all values before aggregating — full shuffle + likely spill. Replace with reduceByKey or aggregateByKey for any associative function. At 5 TB/day this is the most likely single cause of scratch exhaustion.
Sort-merge join, both sides large, no AQE broadcast HIGH Shuffles both input datasets fully. Use broadcast() hint if one side fits in memory. If the join repeats across jobs, bucket the tables on the join key to eliminate the shuffle entirely.
Window function over large, unselectivelypartitioned data HIGH Window functions partition and sort data to scratch. A PARTITION BY clause on a low-cardinality key (e.g., a boolean flag) creates very large partitions that spill aggressively. Partition on a more selective key to distribute the window across more, smaller partitions.
Sort-merge join, AQE enabled, one side < broadcast threshold MEDIUM AQE may convert to broadcast at runtime, reducing or eliminating the shuffle for the smaller side. Not guaranteed — check the query plan in Spark UI. Plan for MEDIUM, hope for LOW.
repartition(N) to increase partition count MEDIUM Full shuffle to redistribute data. Necessary when increasing partition count or rebalancing skewed data. Accept the cost and ensure the downstream stage benefits from the rebalance (e.g., before a wide join or aggregation).
reduceByKey / aggregateByKey LOW Map-side partial aggregation before shuffle. Only partial results cross the network. Standard recommendation for all aggregation jobs.
Bucketed join on matching key and bucket count NEAR ZERO No shuffle Exchange node in the plan — data is already co-located. Best pattern for frequently re-joined dimension or fact tables. Pay once at write time.
Broadcast hash join (small table) NEAR ZERO Small table broadcast to all executors as in-memory hash map. Large table never shuffled. Zero scratch on the join stage itself.
coalesce(N) to reduce partition count NONE Narrow transformation — no shuffle. Use instead of repartition when reducing partition count for output writes.
filter / map / select / withColumn NONE Narrow transformations — no data movement between partitions. No scratch usage from these operations directly.

Estimating your job's amplification before submitting

A rough pre-run estimate using the sizing model:

# Estimate scratch per node before running a job
daily_tb      = 5.0     # your ingest volume
shuffle_amp   = 3.5     # your estimate (2 for well-optimized, 5+ for groupByKey-heavy)
nodes         = 3
scratch_node  = daily_tb * shuffle_amp * 2 / nodes
ceiling       = 15.36   # Config I: 4 drives × 3.84 TB per node

print(f"Estimated scratch per node: {scratch_node:.1f} TB")
print(f"Config I ceiling:           {ceiling:.2f} TB")
print(f"Headroom:                   {'OK' if scratch_node < ceiling else 'EXCEEDS CEILING'}")

If your estimate exceeds the ceiling, review your operators against the table above before submitting. The most common fix is replacing groupByKey with reduceByKey or adding a broadcast hint to the smaller join side.
