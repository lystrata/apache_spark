Tier 0
Physical Constants · Hardware facts · never computed · change only when hardware changes ↑ Top

Hardware facts hard-coded at the top of the calculator script as module-level JavaScript constants. These values never change during a session. If hardware changes (e.g., NVMe drives are added or replaced), these constants must be updated in the source. They are not exposed as sliders.

Development Cluster Hardware
ConstantValueUnitDescription
NODES 3 nodes Physical Proxmox cluster members. All per-node calculations are multiplied by this to get cluster-wide totals.
CORES_NODE 32 cores Physical cores per node. Single-socket AMD EPYC — one NUMA domain. The 32-core vCPU ceiling in the slider is a direct reflection of this value.
NVME_COUNT 7 drives NVMe drives per node. These drives serve dual purpose: Ceph OSDs (user-configurable via slider) and VM scratch space (remainder). Scratch NVMe drives are JBOD/XFS — no hardware RAID. SSDs are separate and not included here.
NVME_TB 3.84 TB Raw capacity of each NVMe drive. Used to compute Ceph raw capacity and scratch space per VM.
SSD_COUNT 3 drives SSD drives per node. Dedicated to Proxmox OS only — configured as ZFS mirror + hot spare. Not part of Ceph or scratch pool.
SSD_GB 0.48 TB (480 GB) Capacity of each SSD. Referenced in the storage layout display only — not used in sizing math.
Note on RAM
RAM per node is not a physical constant in the calculator — it is exposed as the RAM per node slider (range 128–320 GB, step 32 GB, default 320 GB). The physical maximum of 320 GB corresponds to 10 populated DIMMs × 32 GB (per the dev cluster hardware spec — see CLAUDE.md § Hardware Reference). This design allows modelling a partially-populated node without editing the source.

Tier 1
Service Reservations · Fixed overhead per node · consumed before VMs are sized ↑ Top

Before any Spark worker VM can be sized, a fixed block of cores and RAM is consumed on every node by the hypervisor and Ceph services. These are not configurable per-node — they are infrastructure policy constants. The YARN ResourceManager and optional on-cluster services run on a dedicated VM on one node; their cost is amortized (ceiling-divided by NODES) to produce a per-node model.

Per-Node Infrastructure Reservations
ServiceCores / nodeRAM / nodeBasis
Proxmox hypervisor 2c 8 GB Fixed — host OS, QEMU process, live-migration buffers.
ZFS ARC (OS mirror cache) — 8 GB Read cache for the Proxmox OS ZFS mirror (3× 480 GB SSD). Constrained to 8 GB via zfs_arc_max = 8589934592 in /etc/modprobe.d/zfs.conf. 8 GB is generous for an OS-only working set. Included in RESERVE_RAM.
Ceph OSD OSD_NODE × 1c OSD_NODE × 3 GB 1 core + 3 GB per OSD (BlueStore WAL/cache). Scales with the NVMe → Ceph OSDs slider. Default: 5 OSDs → 5c / 15 GB.
Ceph RGW (S3 gateway) 4c 8 GB Fixed — 4 frontend worker threads serving the S3 API on each node.
Ceph MON (monitor) 2c 6 GB Fixed — one Paxos quorum member per node.
YARN ResourceManager VM ⌈2 / NODES⌉ = 1c ⌈4 / NODES⌉ = 2 GB RM runs on a single node (2 vCPU / 4 GB VM). Cost is ceiling-divided across NODES to produce a per-node amortized cost. Always active — not a toggle.
Optional on-cluster services ⌈OPT_SVC_CORES / NODES⌉ ⌈OPT_SVC_RAM / NODES⌉ Airflow (2c/4 GB), History Server (1c/2 GB), Jupyter (2c/4 GB), Log Aggregation (1c/2 GB), Nginx (1c/1 GB). Each runs on the service node; cost amortized across NODES. Default: all off.

User Parameters · Sliders & toggles · dynamic bounds computed from lower tiers ↑ Top

These are the design decisions the user adjusts interactively. Each slider has a static range defined in HTML, but several have dynamic bounds — their effective maximum is recomputed on every recalc() call based on the current state of lower-tier values. When a dynamic max drops below the current slider value, the slider is automatically clamped.

Dynamic Slider Bounds — Detail

Three sliders have bounds that are recomputed on every recalc() call. When the computed maximum drops below the current slider value, the slider is immediately clamped to that maximum and recalc() continues with the clamped value. This prevents the calculator from displaying physically impossible configurations.

Dynamic Bound 1 — RAM per VM (most complex)

The RAM/VM slider maximum is the largest 32 GB-aligned value that fits after subtracting reservations and dividing by VM count. It updates whenever ram_node, nvme_osd_count, vms_per_node, or any service toggle changes. When the maximum changes, the existing slider value is proportionally rescaled so the slider thumb stays at the same relative position — avoiding sudden jumps.

PhaseVariableFormulaNotes
Compute new max _reserveRam 8 + (OSD_NODE×3 + 8 + 6) + ⌈SVC_RAM / 3⌉ Mirrors RESERVE_RAM, computed inline: Proxmox (8 GB) + Ceph OSD/RGW/MON + amortized service VM RAM.
_maxRamVM (raw) ⌊(RAM_NODE − _reserveRam) / VMS_NODE / 32⌋ × 32 Largest 32 GB-aligned value that fits after subtracting reservations and dividing by VM count.
_maxRamVM max(32, _maxRamVM raw) Floor at 32 GB (one DIMM) — slider cannot go below this even at extreme configurations.
Rescale value _min 64 Slider fixed minimum — 2 DIMMs (64 GB).
_pct (current_value − _min) / (old_max − _min) Current slider position as a fraction of the previous range. This fraction is preserved across the max change.
_new round(_min + _pct × (_maxRamVM − _min)) to nearest 32 Same fractional position mapped onto the new range, snapped to the 32 GB DIMM grid.
new_value clamp(_new, _min, _maxRamVM) Final enforced value. If RAM_NODE drops sharply the rescaled result may still exceed the new max — clamp handles this edge case.
Why this runs first
The RAM/VM pre-scale block runs at the very top of recalc(), before any other values are read. This is load-bearing: subsequent executor math reads the (possibly-clamped) RAM/VM value, and SLOT_GB depends on it. Running it after other reads would produce a one-cycle lag where executor memory is computed against a stale RAM/VM value.
Dynamic Bound 2 — vCPUs per VM
VariableFormulaNotes
_infra 2c (Proxmox) + 4c (RGW) + 2c (MON) + OSD_NODE + ⌈SVC_CORES / NODES⌉ Per-node infrastructure cores; mirrors the components of RESERVE_CORES.
_maxVcpu min(32, ⌊(CORES_NODE − _infra) / VMS_NODE⌋) Hard cap of 32 = one NUMA domain (single 32-core socket on dev nodes). Slider is clamped to this value if it exceeds it.

The hard limit of 32 is the NUMA domain boundary — the development node has a single 32-core CPU socket. Exceeding 32 vCPUs means the VM spans beyond the physical socket (impossible on single-socket hardware without hyperthreading), and triggers the NUMA boundary constraint check in Tier 5.

Dynamic Bound 3 — Off-heap size
VariableFormulaNotes
_maxOffheap (raw) ⌊(SLOT_GB − 4) / 2⌋ × 2 Step-2 aligned (slider moves in 2 GB increments). The 4 GB deduction reserves at least 4 GB of OOM headroom before executor overhead is considered. Must be computed after SLOT_GB is known.
_maxOffheap max(0, _maxOffheap raw) Cannot be negative. If SLOT_GB ≤ 4, off-heap is forced to 0 — the slot is too small to accommodate any off-heap allocation.

The 4 GB reserve ensures that even without taking executor overhead into account, at least 4 GB of headroom remains in the slot. The actual OOM margin after executor overhead will typically be smaller — this bound only prevents the slider from being set to a value that is mathematically impossible to satisfy.

YARN NodeManager Overhead (Inside Each Worker VM)

Two Levels of Reservation
Tier 1 reservations happen at the hypervisor (node) level — they reduce how many cores and how much RAM is available to VMs at all. Inside each worker VM, the YARN NodeManager daemon reserves an additional fixed amount before executors are sized. These are different deductions at different levels of the stack.

Tier 4
Second-Order Derived Values · Executor memory chain · blast radius · shuffle parallelism · network transfer ↑ Top

These values are computed from Tier 3 results. They represent the sizing outputs that directly configure Spark — executor memory settings, shuffle parallelism, and network transfer estimates.

Executor Memory Sizing — Core Chain

Key Design Principle
The calculator works backward from available space. It does not accept an executor memory target and check if it fits — instead it computes the maximum memory that fits in the slot, then applies a 5% safety discount to produce the recommended spark.executor.memory value.
Executor Memory — 5-Step Derivation
StepVariableFormulaNotes
1 SLOT_GB EXEC_RAM_VM / EXEC_PER_VM RAM available to one executor including all overhead types. Equal division of VM memory across executors.
2 MAX_EXEC_MEM (SLOT_GB − OFFHEAP) / (1 + OVH_PCT) Constraint rearranged: exec_mem × (1 + overhead) + offheap ≤ slot → exec_mem ≤ (slot − offheap) / (1 + overhead). This is the theoretical maximum before the safety margin.
3 EXEC_MEM ⌊MAX_EXEC_MEM × 0.95⌋ 5% safety margin applied; rounded down to integer GB. This value is written to spark.executor.memory.
4 ACTUAL_TOTAL EXEC_MEM × (1 + OVH_PCT) + OFFHEAP Verification: total container memory actually consumed per slot. Should be ≤ SLOT_GB.
5 OOM_MARGIN SLOT_GB − ACTUAL_TOTAL Remaining headroom in the slot. Negative = executor container exceeds its allocation — OOM on launch. Evaluated against OOM_THRESHOLD in Tier 5.
Memory Slot Anatomy (visual breakdown)
SegmentFormulaColor in UINotes
Executor heap EXEC_MEM GB Blue The value written to spark.executor.memory.
YARN overhead EXEC_MEM × OVH_PCT Yellow JVM overhead above the heap. YARN requires this to be reserved in the container.
Off-heap OFFHEAP GB Purple Direct memory for Parquet column buffers and serialization. Zero if slider is at 0.
OOM margin SLOT_GB − ACTUAL_TOTAL Green / Yellow / Red Headroom before the executor exceeds its container allocation. Color reflects Tier 5 threshold comparison.

Blast Radius

Blast Radius
VariableFormulaNotes
BLAST_PCT 100 / TOTAL_EXEC Percentage of cluster parallelism lost when one executor fails or is killed. Lower is better — more executors means each one carries less proportional weight.

A higher executor count lowers blast radius. With 10 executors, losing one costs 10% of cluster capacity. With 3, losing one costs 33%. This metric argues for more, smaller executors — balanced against OOM risk from narrow slots.

Tier 5
Constraint Evaluations · Warning & error conditions · operating envelope ↑ Top

Boolean checks that run after all derived values are computed. Results are displayed in the Diagnostics panel and Operating Envelope table. Constraints are evaluated on every recalc() call — the UI updates immediately as sliders are moved.

Workload-Dependent Thresholds

Two thresholds are not fixed numbers — they depend on the skew profile and join type selects. This means the same hardware configuration can show different warning states depending on the workload characteristics being modeled.

OOM Threshold — by Skew Profile
Skew ProfileOOM_THRESHOLDRationale
Uniform4 GBEven data distribution; all executors see similar memory pressure.
Moderate skew5 GBSome partitions are larger; hot executors need more headroom.
Heavy skew6 GBSevere key skew causes certain executors to process much more data. Without headroom, OOM kills are likely even with AQE.
Scratch Multiplier — by Join Type
Join TypeJOIN_SCRATCH_FACTORRationale
Broadcast-heavy×1.0Small tables are broadcast to all executors — no shuffle disk required.
Mixed×1.10Combination of broadcast and sort-merge joins adds intermediate spill.
Wide shuffle joins×1.25Full sort-merge joins on large tables generate significant intermediate data on NVMe scratch.

Warning & Error Conditions

Diagnostics Panel — All Conditions
LevelConditionCheck ExpressionMeaning
Error Core overcommit FREE_CORES < 0 VMs are claiming more cores per node than are available after reservations. Jobs will compete for cores with Ceph/Proxmox. Reduce vCPUs/VM or VMs/node.
Error RAM overcommit FREE_RAM < 0 VMs are claiming more RAM per node than is available. Proxmox will swap or refuse to start VMs. Reduce RAM/VM.
Error Executor OOM on launch OOM_MARGIN < 0 Total executor container size exceeds the slot. YARN will kill the executor container before any data is processed. Reduce executor.memory, overhead %, or off-heap.
Warning OOM margin thin 0 ≤ OOM_MARGIN < OOM_THRESHOLD Executor fits, but headroom is below the workload-adjusted threshold. Data skew or large Parquet column buffers could push individual executors into OOM. Threshold is 4/5/6 GB depending on skew profile.
Error NUMA boundary crossed VCPU_VM > 32 vCPU count exceeds the single NUMA domain (32 cores on this single-socket node). Cross-NUMA memory access adds latency to ~50% of memory operations. Must keep vCPUs ≤ 32.
Warning AQE disabled AQE = false Fixed shuffle partition count is unlikely to match real data. AQE dynamically adjusts partitions, handles skew, and coalesces small partitions. Strongly recommended on Spark 3.x.
Warning Low executor density EXEC_PER_VM < 3 Fewer than 3 executors per VM means each executor represents a large fraction of per-VM resources. A single executor failure or OOM has high impact. Consider fewer cores per executor.
Error Ceph capacity critically low CEPH_DAYS < 3 Less than 3 days of ingest retention. Storage will fill before a weekend. Pipeline must run at least ⌈1 / CEPH_DAYS⌉ times per day to avoid data loss.
Warning Limited Ceph retention 3 ≤ CEPH_DAYS < 7 Less than one week of buffer. A stalled Snowflake pipeline could cause overflow. Acceptable for active monitoring but not for unattended operation.
Warning Scratch may be insufficient SCRATCH_TB_PER_VM < DAILY_TB × SHUF_AMP × JOIN_SCRATCH_FACTOR × 0.75 Scratch disk covers less than 75% of the adjusted shuffle target. Spark will spill to disk beyond NVMe capacity — potentially to Ceph via YARN local dirs, which is much slower. Reduce OSD allocation to free NVMe scratch, or reduce shuffle amplification via better partitioning.
OK Configuration healthy TOTAL_EXEC ≥ 6 AND OOM_MARGIN ≥ OOM_THRESHOLD AND FREE_CORES ≥ 0 AND FREE_RAM ≥ 0 AND CEPH_DAYS ≥ 7 All primary constraints are satisfied. The configuration is viable for production workloads.

Shuffle Tuning Parameters — Research-Validated Additions (2026-04-19)
spark.serializer org.apache.spark.serializer.KryoSerializer Smaller serialized shuffle records → less scratch and network I/O. Static best-practice. [2][6]
spark.shuffle.compress true (snappy, default) Compresses map output before writing to scratch. Default on. Typically reduces on-disk shuffle data 40–60%. Calculator input is uncompressed-basis (conservative). [2][8]
spark.shuffle.spill.compress true (default) Same compression applied to reduce-side spill written to scratch. Default on; keep enabled. [2]
spark.io.compression.lz4.blockSize 524288 (512 KB) New — 2026-04-19. Default 32 KB is undersized for large-dataset shuffle. 512 KB yields up to −20% shuffle/spill file size, additive on top of snappy ratio. No performance penalty on NVMe. [6]
spark.shuffle.file.buffer 1m (revised from 64–128 KB) Revised upward — 2026-04-19. In-memory write buffer per map output file. 1 MB reduces disk flush frequency on large shuffle stages. Performance only — no capacity impact. [2][6]
spark.shuffle.memoryFraction 0.30 (raised from 0.20 default) New — 2026-04-19. Fraction of executor heap for shuffle buffers before spilling to scratch. 30% reduces reduce-side spill for join-heavy workloads. [2]
spark.reducer.maxSizeInFlight 96m (raised from 48 MB default) Max fetch size per reduce request. Larger value → fewer round-trips during shuffle fetch. Performance only — no capacity impact. [2][7]
spark.sql.adaptive.enabled AQE toggle (Tier 2) AQE: runtime auto-broadcast conversion, skew-join splitting, partition coalescing. When enabled, AQE can convert wide sort-merge joins to broadcast hash joins at runtime — reducing scratch pressure retroactively. Default true on Spark 3.2+. [5]

2026-04-18 — CEPH_DAYS Formula Revised

Architectural constraints incorporated into Ceph headroom model

Snowflake has no inbound network access. Parquet is transferred by an outbound batch job and deleted only on confirmed delivery. Up to N concurrent Parquet outputs may accumulate on local Ceph. VM OS disks (80 GB per VM, Ceph RBD) were not previously deducted.
