Spark Infrastructure — Glossary

Terms, acronyms, and concepts for the Spark / YARN / Ceph / Airflow stack
ⓘ Overview · how to use this glossary · important disambiguation notes ↑ Top

This glossary covers terminology used across the Spark, YARN, Airflow, Ceph, and monitoring stack. Terms are grouped by system. Use the section headers to navigate to the relevant area.

Two words that mean different things in different contexts:

Executor — in Spark, an executor is a JVM process on a worker node that runs computation. In Airflow, an executor is the plugin that decides how to dispatch pipeline tasks (LocalExecutor, CeleryExecutor). Completely unrelated concepts sharing the same word.

DAG — in Airflow, a DAG is a pipeline definition (a Python file describing a workflow). In Spark's internal model, a DAG is the graph of computation stages the engine builds to execute a query. Same acronym, different scope.

A Spark Core · driver · executor · task · stage · job · shuffle · partition · DataFrame ↑ Top
Apache Spark
A distributed computation engine. Reads data, applies transformations in parallel across a cluster, and writes results. In this environment Spark runs on YARN and processes the daily CSV ingest before loading to Snowflake.
Driver Spark Driver Process
The process that controls a Spark application. It parses user code, builds the execution plan, negotiates resources from YARN, and coordinates executor work. In cluster mode the driver runs inside the cluster; in client mode it runs on the submitting machine (e.g. the bastion VM for Jupyter sessions). The driver is single-threaded by design — it coordinates but does not process data itself.
Executor Spark Executor — not to be confused with Airflow Executor
A JVM process launched on a YARN worker node. Each executor runs tasks assigned by the driver, stores shuffle data, and holds cached RDD/DataFrame partitions in memory. Executor count, CPU cores, and memory are the primary sizing levers in the calculators. ⚠ "Executor" in Airflow means something entirely different — see Section D.
Task
The smallest unit of parallel work in Spark. One task processes one partition of data on one executor core. A stage may contain thousands of tasks running simultaneously across all executors.
Stage
A set of tasks that can run in parallel without a shuffle between them. A Spark job is divided into stages at every point where data must be redistributed across executors (a shuffle boundary). Stages run sequentially; tasks within a stage run in parallel.
Job
A complete unit of Spark work, triggered by an action (writing to disk, counting rows, etc.). A job is composed of one or more stages. The Spark History Server shows one entry per job.
Shuffle
The redistribution of data across executor partitions — necessary for operations like joins, aggregations, and sorts that require rows from different partitions to be co-located. Shuffles write intermediate data to disk and transfer it over the network, making them the most expensive operation in most Spark pipelines. Shuffle amplification: the ratio of shuffle write volume to input data size. A 2 TB ingest with 3× amplification generates 6 TB of shuffle data — a key input to scratch disk sizing.
Partition
A chunk of data processed by a single task. The number of partitions determines parallelism — more partitions means more tasks running in parallel (up to available executor cores). Too few partitions underutilizes the cluster; too many creates overhead from task scheduling.
RDD Resilient Distributed Dataset
The original Spark data abstraction — an immutable, distributed collection of objects. Largely superseded by DataFrames for most workloads but still used internally by Spark and in some low-level operations.
DataFrame
A distributed table with named columns and a schema, built on top of RDDs. The primary API for structured data processing. Spark optimizes DataFrame operations through the Catalyst query optimizer before executing them.
SparkSession
The entry point to Spark functionality in code. Creating a SparkSession starts the driver process and connects to the cluster. One SparkSession per application.
spark-submit
The command-line tool used to submit a Spark application to a cluster. Accepts parameters for executor count, memory, CPU, deployment mode, and the application JAR or Python file. Airflow uses spark-submit internally when the SparkSubmitOperator fires a production job.
Spark History Server
A web UI that displays metrics and event logs for completed Spark jobs. Reads event log files written to a persistent location (S3/Ceph RGW in this environment) after a job finishes. Used for post-job performance analysis and debugging.

B YARN & Cluster Resources · ResourceManager · NodeManager · client mode · cluster mode · container ↑ Top
YARN Yet Another Resource Negotiator
The cluster resource manager that Spark runs on in this environment. YARN receives job submissions, allocates CPU and memory containers across worker nodes, and manages running applications. It is the scheduling layer between Spark and the physical hardware.
ResourceManager RM
The YARN master process. Runs on one node and accepts job submissions (spark-submit sends to port 8032). Tracks available resources across all NodeManagers and allocates containers to applications. The YARN web UI runs on port 8088 and is proxied through Nginx on the bastion VM.
NodeManager NM
The YARN worker process that runs on every node where executors will live. Receives container launch requests from the ResourceManager, starts executor JVMs, and reports resource usage back. One NodeManager per worker VM.
ApplicationMaster AM
A per-job process launched by YARN when a Spark job is submitted in cluster mode. The ApplicationMaster negotiates executor containers from the ResourceManager on behalf of the Spark driver. In cluster mode, the Spark driver runs inside the ApplicationMaster process.
Container
A resource allocation unit in YARN — a fixed amount of CPU (vCores) and memory assigned to one process on one NodeManager. Each Spark executor runs in its own container. The container size determines executor memory and core count.
vCore
A virtual CPU unit in YARN's resource model. YARN tracks CPU availability in vCores; each executor container requests a number of vCores. In this environment one vCore maps to one vCPU on the VM.
Client Mode
A Spark deployment mode where the driver process runs on the machine that submitted the job — outside the cluster. Used for interactive development (Jupyter on the bastion VM). Driver logs appear locally, making debugging easier. The driver must stay alive for the job duration; if the submitting machine dies, the job dies. In this environment: Jupyter notebooks run in client mode — Spark driver on bastion VM, executors in the YARN cluster.
Cluster Mode
A Spark deployment mode where the driver runs inside the cluster as an ApplicationMaster. Used for production jobs submitted by Airflow. The submitting process (Airflow's spark-submit) exits once the job is handed off; the job continues independently inside YARN. In this environment: Airflow production DAGs submit in cluster mode — driver and executors all run inside the YARN cluster.

C Memory & Performance · executor memory · overhead · OOM · shuffle amplification · spill · broadcast join ↑ Top
Executor Memory --executor-memory
The JVM heap allocated to each executor container. This is the primary parameter tuned in the Spark calculators. Larger executor memory means more data fits in RAM before spilling to disk, but fewer executors fit per node.
Memory Overhead
Non-heap memory reserved alongside each executor's JVM heap — covers JVM internals, off-heap buffers, and network memory. Defaults to 10% of executor memory (minimum 384 MB). YARN enforces the total container size as executor-memory + overhead; exceeding it kills the container with an OOM.
Off-Heap Memory
Memory allocated outside the JVM heap, managed directly by Spark's Tungsten engine. Used to avoid JVM garbage collection pauses on large datasets. Configured separately from executor memory and counted against the container total.
OOM Out of Memory
An executor killed because it exceeded its YARN container memory limit. Typically caused by data skew (one partition much larger than others), insufficient executor memory, or memory overhead set too low. The calculators include an OOM margin warning when the safety buffer falls below 4 GB.
Spill
When Spark cannot fit intermediate shuffle or aggregation data in executor memory, it writes the overflow to local disk (scratch space). Spill degrades performance but prevents OOM failures. The calculators size NVMe scratch space to accommodate expected spill volume.
Shuffle Amplification
The ratio of shuffle write volume to input data size. A pipeline reading 2 TB with a shuffle amplification of 3× writes 6 TB of intermediate shuffle data. This is the primary driver of scratch disk sizing. Highly join-heavy or aggregation-heavy pipelines have higher amplification than simple filter/transform jobs.
Broadcast Join
A join optimization where a small table is copied in full to every executor, avoiding a shuffle entirely. Spark applies this automatically when one side of a join is below a configurable threshold (default 10 MB). Dramatically reduces shuffle volume for fact-dimension joins.
Speculation
A Spark feature that re-launches slow-running tasks on different executors in parallel. If the re-launched copy finishes first, the original is killed. Useful when hardware variability causes stragglers, but can increase resource consumption.
Data Skew
A condition where data is unevenly distributed across partitions — a small number of partitions are much larger than the rest. Skew causes some tasks to run far longer than others (stragglers) and can trigger OOM on the oversized partitions. Common with joins on non-uniform key distributions.

D Airflow & Orchestration · DAG · operator · scheduler · executor · LocalExecutor · CeleryExecutor ↑ Top
In Airflow, DAG and Executor have meanings specific to Airflow's pipeline model — distinct from the same words in Spark's execution model. See the Overview section for the full disambiguation.
Apache Airflow
A workflow orchestration platform. Pipelines are defined as Python code (DAGs). The Airflow scheduler monitors DAG definitions, fires tasks on schedule, retries failures, and provides a web UI for monitoring. In this environment it is the conductor for the daily ETL: trigger ingest → submit Spark job → load to Snowflake.
DAG Directed Acyclic Graph — Airflow context
An Airflow pipeline definition written in Python. A directed graph of tasks where edges represent dependencies and there are no cycles (a task cannot depend on itself, directly or indirectly). Airflow's scheduler reads DAG files, determines which tasks are ready to run, and dispatches them. ⚠ In Spark's internal model, "DAG" refers to the graph of computation stages. Same acronym, different context.
Task Airflow Task — not to be confused with Spark Task
A single unit of work within an Airflow DAG — for example, "run this spark-submit command" or "check if this S3 file exists." Defined using an Operator. Tasks have dependencies on other tasks within the same DAG. ⚠ A Spark Task is the smallest unit of parallel computation inside a Spark job. These are unrelated concepts.
Operator
The type of an Airflow task — defines what the task does. Common operators in this environment: SparkSubmitOperator (submits a Spark job to YARN), S3KeySensor (waits for a file to appear in S3/Ceph), BashOperator (runs a shell command), SnowflakeOperator (runs a SQL statement in Snowflake).
Scheduler Airflow Scheduler
The Airflow process that continuously parses DAG files, evaluates task dependencies, and queues tasks that are ready to run. Always-on; typically the most resource-intensive Airflow component. One scheduler process per Airflow deployment.
Executor Airflow Executor — not to be confused with Spark Executor
The plugin that determines how Airflow dispatches queued tasks. The executor type is a deployment-level configuration choice — it does not affect individual DAG logic. ⚠ Nothing to do with Spark executors. Airflow Executor = "how tasks are dispatched". Spark Executor = "the JVM process running computation on a worker node".
LocalExecutor
An Airflow executor that runs tasks as subprocesses on the same machine as the Airflow scheduler. No additional infrastructure required. Sufficient for moderate task concurrency. The assumed executor type for this deployment — appropriate for a daily batch ETL workload where Spark handles all heavy computation.
CeleryExecutor
An Airflow executor that distributes tasks to separate worker processes via a message queue (Redis or RabbitMQ). Designed for high-concurrency workloads or deployments where tasks must run on different machines. Requires Redis as an additional dependency. Not needed for this deployment unless task concurrency demands it.
DAG Run
One execution instance of a DAG — the record of a single pipeline run, including start time, end time, and status of every task. Airflow's UI shows DAG runs as rows in the grid view.
Task Instance
One execution of a specific task within a specific DAG Run. Has its own status (queued, running, success, failed, retrying) and its own log. The lowest-level entity in Airflow's execution model.
SparkSubmitOperator
An Airflow operator that runs spark-submit to submit a Spark job to a cluster. Configured with executor count, memory, the target YARN ResourceManager address, and the application JAR or script. Blocks until the job completes, then reports success or failure back to the DAG.

E Storage & Ceph · OSD · MON · MGR · RGW · S3 · replication · exaBGP ↑ Top
Ceph
An open-source distributed storage system providing object, block, and file storage from a shared pool of drives. In this environment Ceph runs across all three Proxmox nodes, using the NVMe drives as OSDs. Spark reads and writes data via Ceph's S3-compatible gateway (RGW).
OSD Object Storage Daemon
The Ceph process that manages one physical drive. Each NVMe allocated to Ceph runs one OSD. OSDs store data chunks, replicate them to peer OSDs, and handle recovery when a drive or node fails. OSD count per node is a key input to the calculators — it determines both Ceph usable capacity and the NVMe drives left for Spark scratch.
MON Monitor
The Ceph process that maintains the authoritative cluster map and manages quorum. A 3-node Ceph cluster runs 3 MONs — quorum requires 2 of 3 to be alive, so the cluster tolerates one MON failure. MONs do not store data; they store metadata about where data lives.
MGR Manager
The Ceph process that provides monitoring, orchestration, and plugin functionality (dashboard, Prometheus metrics exporter). Runs active/standby — one active MGR at a time, with a standby on another node for failover.
RGW RADOS Gateway
Ceph's S3-compatible object storage gateway. Exposes Ceph storage via the S3 API so that any S3-aware client (Spark, Airflow, AWS CLI, etc.) can read and write data without knowing about Ceph internals. In this environment RGW is the interface through which Spark reads CSV files and writes processed output.
S3 Simple Storage Service
Amazon's object storage protocol — a standard HTTP API for storing and retrieving named objects in buckets. Ceph RGW speaks the S3 protocol, so tools written for AWS S3 work against Ceph without modification. "S3" in this document always refers to Ceph RGW unless AWS is explicitly mentioned.
Replication Factor
The number of copies Ceph stores of each data chunk across OSDs. Set to 3 in this environment — every piece of data exists on 3 different OSDs across (ideally) 3 different nodes. One full node can fail without data loss. The tradeoff: usable capacity = raw capacity ÷ replication factor.
RADOS Reliable Autonomic Distributed Object Store
The underlying Ceph storage layer. All higher-level interfaces (RGW, RBD block storage, CephFS) are built on top of RADOS. When you read or write via the S3 API, RADOS handles the actual data placement across OSDs.
exaBGP
A BGP route injection tool used to announce a floating IP address for the Ceph RGW endpoint. Whichever node is healthy advertises the floating IP via BGP; if that node fails, another node takes over the announcement. Clients (including the bastion VM and Spark executors) always use the floating IP — failover is transparent.
Scratch Space
Local NVMe capacity on each worker node reserved for Spark shuffle and spill data. Not Ceph-replicated — it is ephemeral, node-local storage. When a Spark job shuffles data it writes to scratch, transfers between nodes over the network, and discards after the job completes. Scratch sizing is driven by shuffle amplification and concurrent job count.

F Infrastructure & Monitoring · Proxmox · NUMA · ZFS · Prometheus · Grafana · Loki · Promtail ↑ Top
Proxmox VE Proxmox Virtual Environment
An open-source hypervisor platform based on KVM (for VMs) and LXC (for containers). The physical nodes in this environment run Proxmox, which hosts the YARN executor VMs and Ceph service VMs. Proxmox also manages the cluster network and storage pools.
Hyperconverged Infrastructure HCI
An architecture where compute (CPU/RAM for VMs) and storage (Ceph OSDs) run on the same physical nodes, sharing the same hardware. The 3-node production cluster is hyperconverged — the same servers run YARN executor VMs and Ceph OSDs simultaneously. The tradeoff is resource contention between compute and storage workloads.
NUMA Non-Uniform Memory Access
A hardware architecture in multi-socket servers where each CPU socket has its own local memory bank. Accessing memory on a remote socket (cross-NUMA) is slower than local access. In this environment the production nodes have 2 CPU sockets (2 NUMA domains × 32 cores each). VMs pinned to one NUMA domain get predictable memory latency; VMs spanning both domains may pay a cross-NUMA penalty.
ZFS
A filesystem and volume manager used on the Proxmox hosts to store VM disk images. ZFS provides checksumming, compression, and snapshots. The 3× 480 GB SSDs per node run ZFS (mirrored pair + hot spare) for the OS and VM disk storage pool.
ARC Adaptive Replacement Cache
ZFS's in-memory read cache. ARC aggressively consumes available RAM to cache frequently accessed data — beneficial for ZFS but it competes with Spark executor memory. In this environment ZFS ARC is capped at 8 GB per node (zfs_arc_max=8589934592) via the Proxmox configuration to protect executor memory headroom.
vCPU
A virtual CPU assigned to a VM. Maps to one hyperthread on the physical host. The calculators track vCPU allocation per node across all VMs to ensure physical core counts are not overcommitted.
Prometheus
An open-source metrics collection and alerting system. Scrapes numeric time-series metrics from exporters running on each host at a configurable interval (15 seconds in this environment). Stores metrics in its TSDB (time-series database). Grafana uses Prometheus as a data source for dashboards.
TSDB Time Series Database
Prometheus's on-disk storage format for metrics data. Highly compressed — approximately 1.5 bytes per sample. At 15-second scrape intervals across ~12,000 series, 90-day retention requires roughly 10–15 GB of disk.
Grafana
A dashboarding and visualization platform. Connects to data sources (Prometheus for metrics, Loki for logs) and renders graphs, tables, and alerts. In this environment a new Grafana instance on the monitoring cluster will consolidate infrastructure dashboards for all Proxmox nodes, YARN, Ceph, and bastion services.
Loki
A log aggregation system developed by Grafana Labs. Receives log streams from Promtail agents, indexes them by labels (host, service, log level), and makes them searchable through Grafana's log explorer. Unlike Elasticsearch, Loki does not index log content — it indexes metadata only, keeping resource usage low.
Promtail
The Loki log shipping agent. Runs on each host, tails configured log files and the systemd journal, attaches labels, and forwards log lines to a Loki instance. Lightweight: approximately 100 MB RAM per host. One Promtail agent will run on the bastion VM and on each of the three Spark cluster nodes.
node_exporter
A Prometheus exporter that exposes Linux host metrics: CPU utilization, memory usage, disk I/O, network throughput, filesystem capacity, and more. Runs as a lightweight daemon on every monitored host (~800 time series per host). The primary source of raw hardware visibility in Grafana dashboards.
pve_exporter
A Prometheus exporter for Proxmox VE. Exposes VM status, CPU and memory usage per VM, storage pool utilization, and cluster node health via the Proxmox API. Adds approximately 600 time series per Proxmox node.
Bastion VM
In this environment, the VM on the second Proxmox cluster that hosts all user-facing services: Airflow, Spark History Server, JupyterHub, and Nginx. Users interact with the Spark cluster exclusively through the bastion VM — they never SSH directly into cluster nodes. "Bastion" traditionally refers to a hardened jump host; here it extends to cover the full service layer.
