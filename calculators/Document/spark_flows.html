StepDetail
Schedule firesAirflow scheduler evaluates the DAG on its cron schedule and queues tasks that are ready.
Task 1 — Spark submitSparkSubmitOperator runs spark-submit targeting YARN ResourceManager port 8032. Job submits in cluster mode — driver and executors all run inside the cluster. Airflow blocks until completion.
YARN allocationResourceManager assigns executor containers across the three nodes. Each container gets the configured vCores and memory from the sizing calculator output.
Spark jobExecutors read compressed CSV files from Ceph S3, apply transformations, shuffle intermediate data to local NVMe scratch, and write output back to Ceph S3. Event logs are written to the S3 event log bucket.
Task 2 — Snowflake loadOn Task 1 success, SnowflakeOperator fires a COPY INTO statement loading the output from S3 into Snowflake.
CompletionAirflow marks the DAG run complete (green). Failure at any task triggers the retry policy and sends an alert.

Client mode vs cluster mode: In development, the Spark driver runs on the bastion VM (client mode) — output and errors appear directly in the Jupyter notebook. In production (Flow A), the driver runs inside the cluster (cluster mode) — Airflow submits and waits; the cluster is self-contained.
Touch pointDetail
JupyterHubBrowser-based notebook environment on the bastion VM. Developer writes and runs Spark code without SSH access to the cluster.
Spark DriverStarts on the bastion VM when the notebook creates a SparkSession. Negotiates executor containers from YARN, then coordinates task execution.
YARN UI via NginxDeveloper opens the YARN UI in a browser tab to watch live executor allocation and task progress. Nginx rewrites cluster-internal links so all URLs resolve to the bastion.
History ServerAfter the job completes, developer opens the History Server to review stage timings, executor memory usage, shuffle volume, and skew. Used to tune before promoting to a production DAG.
Ceph S3Executors read source data and write results to S3. The developer uses the same data paths as production.

Failure independence: The monitoring cluster and the Spark cluster run on separate Proxmox hardware. If the Spark cluster fails, Grafana continues running and shows the cluster going dark. If the bastion VM fails, the cluster keeps processing — no new job submissions until the bastion recovers, but data already in flight completes safely.
