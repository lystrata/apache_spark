Revisions · Tool change history
2026-05-05 — Vendor config baseline alignment (Ksolves Spark/YARN v1.0)

Updated defaults to match the vendor's authoritative configuration (Ksolves_Spark_YARN_Config_v1.0.pdf, 2026-05-04) and the 3-node hardware finalization on 2026-05-05: RAM/node default 384 → 320 GB (10× 32 GB DIMMs per node); Daily ingest default 5 → 1.5 TB (vendor's measured DEV volume, not the previously-assumed full-prod 4 TB). Vendor reference values worth knowing when configuring this tool: spark.executor.cores = 8, spark.executor.memory = 28g, spark.sql.shuffle.partitions = 4096, repartition formula for any compressed table > 1 GB: target_partitions = max(24, ceil(compressed_csv_mb / 50)) — the Airflow DAG injects compressed_mb per table at runtime. Deferred to a follow-up: NUMA-pinning toggle (currently advisory via the >32-cores warning) and AQE on/off toggle (currently the model assumes AQE-on with the vendor's 4096 partitions as a soft default). When those land they belong in Section A and Section C respectively.

2026-05-04 — Section B: Infrastructure Reservations

Added a new Section B — Infrastructure Reservations with three cards: B.1 Per-Node Fixed Services (Proxmox · ZFS ARC · Ceph MON / MGR / OSD per drive / RGW), B.2 Cluster-Singleton VMs (YARN ResourceManager and Spark History Server, each with a count slider 1 → NODES), and B.3 Remote Host Services (node_exporter and Promtail default-on; Ceph Dashboard / MDS and two custom slots toggleable). The hardware strip gained two new chips: Reserved / Node and Available / Node. Section D executor math now reads from AVAIL_R_NODE = RAM_NODE − fixed − averaged_singleton instead of raw RAM. Singletons use averaged-per-node cost for the headline math; B.2 also reports a worst-case node figure (all singletons on one node) as a fit-check, with an error diagnostic if that exceeds remaining capacity. Existing close-section / close-subsection handlers updated to call closeSub(this) per the project-wide position-preserving close rule. Sections previously labeled B / C / D renumbered to C / D / E.

2026-04-18 — Initial Release

General-purpose cluster sizing tool derived from the dev and production calculator math. All hardware parameters (nodes, cores, RAM, NVMe count, NVMe size, OSD count, RAID type) are exposed as inputs rather than baked-in constants, allowing any cluster configuration to be modelled. Key addition over the fixed calculators: the Shuffle Amplification Ceiling metric, which directly answers "how complex a query can this hardware sustain at the configured ingest volume" without requiring manual scratch calculation.
