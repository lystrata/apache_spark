1. Overview & Purpose

This calculator sizes a 3-node Proxmox hyperconverged development cluster running Apache Spark via YARN, Ceph for distributed storage (with S3 access via RGW), and a daily ETL pipeline that reads raw CSV data, converts it to Parquet, stages it to Ceph RGW, then loads it into Snowflake via COPY INTO.

Hardware is fixed: 3 nodes, each with 32 cores (single AMD EPYC socket), 320 GB RAM (10 × 32 GB DIMMs), 7× 3.84 TB NVMe, and 3× 480 GB SSD (two-drive ZFS mirror + hot spare). The calculator models how those fixed resources are divided between infrastructure services, Spark worker VMs, and Spark executors.

Key difference from production: the development cluster has a single CPU socket per node, which means a maximum of 32 vCPUs per VM with no cross-NUMA traffic. The production cluster has 2 sockets (64 cores per node) where the NUMA boundary constraint is more operationally significant. On dev, NUMA pinning has limited effect since there is only one memory domain per node.

The calculator is organised into six collapsible sections (A–F). Each section can be independently opened or closed to focus on a specific concern. When a section is collapsed and an upstream slider change affects one of its key outputs, an amber UPDATED badge appears on the section header, alerting you that the section’s values have changed without needing to open it first. Opening the section clears the badge.

All sliders are live — every change immediately recalculates all outputs, bar charts, metric tiles, and warnings without any page reload or submit step.

2. Header — Scenario Management & Hardware Constants

Save & Reset

Save writes the current slider positions to browser localStorage under the key spark_calc_dev. The next time you open the calculator in the same browser, the session is automatically restored to the saved state. Reset clears that localStorage entry and returns every slider to its factory default value.

Named Scenarios

The scenario system lets you preserve and compare multiple configurations:

  • Save Scenario — prompts for a name, then stores a named snapshot in a separate localStorage key (spark_calc_dev_scenarios). Any number of named scenarios can coexist.
  • Load — applies a saved scenario’s slider values and immediately calls recalc(). Loading a scenario does not overwrite the session continuity save — you must click Save explicitly if you want the session to persist.
  • View JSON — opens a popup showing the raw JSON for the selected scenario. Useful for copying settings to share with a colleague or pasting into a ticket.
  • Delete — removes the named scenario from localStorage. The current slider state is not affected.

Hardware Constants Bar

The constants bar at the bottom of the header displays the fixed physical parameters for the cluster: NODES = 3, CORES/NODE = 32, NUMA DOMAINS = 1 (single socket — the primary difference from the production cluster), SSDs/NODE = 3×480 GB (ZFS mirror + hot spare), and NVMe/NODE = 7×3.84 TB. RAM/NODE reflects the current RAM slider value.

Section UPDATED Badges

Each collapsible section (A–F) watches a set of key output values. When a section is closed and a slider change causes any of those outputs to change, an amber UPDATED badge appears on the section header and the header briefly pulses to draw attention. Opening the section clears the badge. This mechanism means you can focus on one section at a time without losing track of downstream effects from slider changes made elsewhere.

Display Theme

The ☼ Light / ☾ Dark button at the right end of the action bar toggles between dark mode (default) and the warm-tone light theme. Your preference is stored in localStorage under the key spark_theme and is restored automatically on your next visit. The setting is shared between the development and production calculators — switching theme in one applies it to both.

3. Diagnostics & Warnings

The Diagnostics card is positioned at the top of the page so configuration errors are immediately visible. It re-evaluates on every slider change. The possible states are:

Indicator Severity Meaning & Remediation
✕ Core overcommit Error VMs are claiming more cores than the node has available after reservations. Reduce vCPUs/VM or VMs/node until the core budget bar shows headroom.
✕ RAM overcommit Error VMs are claiming more RAM than the node has after infrastructure reservations. Reduce RAM/VM. The RAM/VM slider prevents exceeding the hardware ceiling, but changes to OSD count or enabling on-cluster services can push a previously valid value into overcommit.
✕ Executor OOM Error The calculated executor.memory × (1 + overhead) + offheap exceeds the slot size. YARN will kill the container on startup. Reduce executor cores (to fit more, smaller executors) or reduce the overhead fraction.
⚠ OOM margin thin Warning The free margin per executor slot is below the active threshold. The threshold is skew-profile-dependent: 4 GB for Uniform, 5 GB for Moderate skew, 6 GB for Heavy skew (set in Section D Workload Sizing). Data skew, large Parquet files, or shuffle spill can push native memory over the YARN container limit. Increase RAM/VM, reduce off-heap allocation, or select a lower skew profile if the workload does not require it.
✕ NUMA boundary crossed Error vCPUs per VM exceeds 32. On the development cluster (single socket), this cannot occur physically — the vCPU slider is hard-capped at 32. The check exists for model consistency.
⚠ NUMA pinning disabled Warning Recommends enabling NUMA pinning for lower memory latency. On dev (single socket), the practical benefit is minimal, but enabling it is a best practice for consistency with production behaviour.
⚠ AQE disabled Warning Shuffle partitions will be a fixed count. For variable-size daily loads, fixed partitions are often too high (small datasets) or too low (large datasets). Enabling Adaptive Query Execution allows Spark to coalesce partitions at runtime.
⚠ Low executor density Warning Fewer than 3 executors per VM. With only 1–2 executors per VM, a single skewed task can stall the entire node. Aim for at least 3 executors per VM by reducing spark.executor.cores or increasing vCPUs/VM.
✕ Ceph capacity critically low Error Ceph usable capacity divided by daily ingest volume is less than 3 days. Increase OSD count, reduce the replication factor (with caution), or increase NVMe-to-OSD allocation.
⚠ Limited Ceph retention Warning Ceph retention is less than 7 days at the current ingest rate. Consider adding more OSDs or reviewing data lifecycle policies.
⚠ Scratch disk may be insufficient Warning Scratch TB per VM is less than 75% of the adjusted shuffle volume: daily_TB × shuffle_amp × join_scratch_factor. The join factor is 1.00 (broadcast-heavy), 1.10 (mixed), or 1.25 (wide shuffle joins), set in Section D. Reduce OSD count to free more NVMe for scratch, choose a smaller join-type factor, or reduce shuffle amplification.
✓ Configuration looks healthy OK All checks pass. No overcommit, adequate OOM margin at the active skew threshold, sufficient Ceph retention, adequate scratch space after join-type adjustment, and healthy executor density.

Below the warning list, the Operating Envelope panel shows the current configuration’s key metrics colour-coded against thresholds: green (safe), amber (marginal), red (unsafe). The active skew profile and join type are shown as context so you can see which constraint thresholds are in effect at a glance.

4. Section A — Infrastructure Provisioning

Section A covers the physical and virtual machine topology inputs. The left column contains the Infrastructure Parameters card where you set RAM, VM count, core allocation, storage split, and optional services. The right column shows the Resource Budget card, which reflects the immediate consequences of those inputs as stacked bar charts and metric tiles.

Infrastructure Parameters

These sliders and toggles define the physical and virtual machine topology. They feed the reservation math and set the ceiling for all executor calculations. Every change here propagates immediately through the entire calculation chain.

Slider / Toggle Range What it controls & what it affects
RAM per node (GB) 128–768 GB, step 32 One step equals one 32 GB DIMM. Sets the total memory budget for the node. Dynamically recalculates the RAM/VM slider’s maximum ceiling every time it moves. At 256 GB (8 DIMMs) the dev cluster operates at 66.6% of peak memory bandwidth per HPE DDR5 DIMM population rules. Increasing toward 320 GB (10 DIMMs, the dev cluster maximum) improves bandwidth utilization.
VMs per node 1–2 How many Spark worker VMs share each node’s resources. 1 VM gets the full single socket with all available cores and RAM. 2 VMs each get half the available cores and half the RAM, and also halves the scratch NVMe allocation per VM. Use 2 VMs to isolate different workloads or Spark versions.
vCPUs per worker VM 8–32, step 2 Maximum is 32 — the single-socket boundary for this cluster. The usable vCPU count for executors is this value minus 2 (reserved for guest OS + YARN NodeManager). Directly drives EXEC_PER_VM = floor((vCPUs−2) ÷ exec_cores) and feeds the core budget bar.
RAM per worker VM (GB) 64 GB to dynamic max, step 32 The dynamic maximum is computed from current RAM/node, OSD count, active services, and VMs/node — you cannot set this higher than the hardware allows. The slider ceiling auto-adjusts whenever any of those values change. If an existing value exceeds the new ceiling it is proportionally rescaled to the same relative position. Usable RAM for executors = RAM_VM − 2 GB (YARN NodeManager overhead).
NVMe drives → Ceph OSDs 0–7, step 1 Splits the 7 NVMe drives between Ceph distributed storage and local Spark scratch. Each OSD contributes 3.84 TB to the Ceph raw pool and reserves 1 core + 3 GB RAM from the node budget. The remaining drives (7 − OSD_count) are divided across worker VMs as scratch space for shuffle spill. Setting to 0 disables Ceph storage entirely.
Ceph replication factor 2–3× Determines how many copies of each data block Ceph maintains. Ceph usable = raw ÷ rep_factor. 3× means 3 copies of every byte — safe against 2 simultaneous drive failures but uses 3× the raw capacity. 2× recovers more usable space but tolerates only 1 failure. Only affects storage capacity and days-of-ingest metrics — no CPU or RAM impact.
NUMA pinning on/off toggle On the development cluster (single socket per node), NUMA pinning has limited operational benefit since there is only one memory domain. The toggle controls warning text — disabling it triggers a yellow advisory. It does not change any numeric calculation. Enable it as a best practice to ensure consistent memory access patterns and to match production behaviour.
AQE enabled on/off toggle Adaptive Query Execution (spark.sql.adaptive.enabled). When ON: spark.sql.shuffle.partitions is treated as a ceiling and Spark coalesces small partitions at runtime based on actual data sizes — critical for variable ETL loads. When OFF: the calculated partition count is used as a fixed value. Disabling triggers a yellow warning.

On-Cluster Optional Services

Below the main sliders, a group of individual service checkboxes controls which optional services run on-cluster rather than on a remote host. YARN ResourceManager is always reserved at 2 vCPUs / 4 GB total, amortized across all 3 nodes (÷3 per node, ceiling-divided). Each optional service adds its own amortized per-node cost:

Airflow 2 vCPU / 4 GB ⌈2 ÷ 3⌉ = 1 vCPU, ⌈4 ÷ 3⌉ = 2 GB per node
Spark History Server 1 vCPU / 2 GB ⌈1 ÷ 3⌉ = 1 vCPU, ⌈2 ÷ 3⌉ = 1 GB per node
Jupyter 2 vCPU / 4 GB ⌈2 ÷ 3⌉ = 1 vCPU, ⌈4 ÷ 3⌉ = 2 GB per node

The amortized per-node costs are deducted from each node’s core and RAM budget, which in turn lowers the ceiling for the RAM/VM and vCPUs/VM sliders. Enabling multiple services reduces the headroom available to worker VMs.

Ceiling effect from services: Each service you enable shrinks the available-for-VMs pool. The RAM/VM slider ceiling adjusts immediately; if your current RAM/VM value now exceeds the new ceiling it will be proportionally clamped downward. Always check Section A’s Resource Budget after enabling or disabling services.

Resource Budget

The Resource Budget card provides a visual summary of how cores and RAM are consumed across the budget categories on each node. Both bars update in real time on every slider change.

Core Budget Bar

The core bar is divided into four segments: Proxmox (dark, always 2 cores) → Ceph + RGW + services (amber, scales with OSD count and enabled services) → Worker VMs (blue, vCPUs/VM × VMs/node) → Unallocated headroom (green if positive, red if overcommit). The free/overcommit segment is what the Diagnostics card monitors for the core overcommit error.

RAM Budget Bar

Same four-segment structure for RAM: Proxmox (8 GB) → Ceph + RGW + services (scales with OSDs and enabled services) → Worker VMs (RAM/VM × VMs/node) → Unallocated (green) or Overcommit (red).

Metric Tiles

  • Total VMs: VMs/node × 3 nodes. The cluster-wide Spark worker count.
  • Free cores: cores not allocated to any reservation or VM. Green = headroom, red = overcommit.
  • Free RAM: RAM not allocated. Green = headroom, red = overcommit.
  • Executors/VM: how many Spark executors fit in one worker VM given the current vCPUs/VM and exec_cores settings from Section C.
Overcommit indicator: Free cores/RAM tiles turning red means you have over-allocated. Spark will not be able to acquire YARN containers and jobs will queue indefinitely or fail. The Diagnostics card at the top will also flag this with a ✕ error. Reduce vCPUs/VM, RAM/VM, or VMs per node to restore headroom.

6. Section C — Executor Sizing

Section C concentrates all executor-related configuration and visualisation. The left column has the Spark Executor Parameters card and Executor Math card. The right column has the Executor Memory Layout visual and the Core Sensitivity ±1 comparison table.

Spark Executor Parameters

These three sliders control how Spark executors are sized within each worker VM. They directly determine job throughput, GC pressure, and YARN container stability.

Slider Range What it controls & what it affects
spark.executor.cores 2–10, step 1 Controls how many vCPUs each executor uses. Fewer cores = more executors per VM = smaller memory slot each (faster GC, lower heap pressure). More cores = fewer executors = larger memory slot (better for large joins). The current default of 4 gives a good balance between executor count and heap size on this cluster’s RAM-per-core ratio. Directly drives EXEC_PER_VM = floor((vCPUs−2) ÷ exec_cores).
Overhead fraction (%) 8–20%, step 1 Maps to spark.executor.memoryOverheadFactor. The JVM reserves this fraction of executor heap for native structures: thread stacks, NIO buffers, and internal metadata. If set too low, YARN kills the container when native memory exceeds the container limit — this appears as a sudden executor loss without a Java exception. Formula: MAX_EXEC_MEM = (SLOT_GB − offheap_GB) ÷ (1 + overhead_fraction).
Off-heap size (GB) 0–16 GB, step 2 Maps to spark.memory.offHeap.size. Carves a fixed GB out of the executor slot for Spark’s vectorized Parquet reader and serialization buffers — this memory lives outside the JVM heap and is not garbage-collected. Every GB here directly reduces the heap available to executor.memory. Set to 0 to disable off-heap entirely (spark.memory.offHeap.enabled = false).
SLOT_GB = (RAM_VM − 2) ÷ EXEC_PER_VM
MAX_EXEC_MEM = (SLOT_GB − offheap_GB) ÷ (1 + overhead_fraction)
EXEC_MEM = floor(MAX_EXEC_MEM × 0.95) // 5% safety margin
OOM_MARGIN = SLOT_GB − (EXEC_MEM × (1 + overhead) + offheap)
Understanding executors and off-heap memory

When Spark runs a job, YARN launches executor processes on your worker VMs — one or more per VM. Each executor is a self-contained JVM (Java Virtual Machine) process that gets its own slice of RAM and CPU cores. Tasks — the individual units of computation (“process partition 47 of the CSV”) — are handed to executors, which run them in parallel across their allocated cores. 

Worker VM  (e.g. 320 GB RAM, 32 vCPUs)
├── Executor 1  (e.g. 120 GB RAM, 5 cores)  ← JVM process
├── Executor 2  (e.g. 120 GB RAM, 5 cores)  ← JVM process
└── OS + NodeManager overhead

Each executor runs tasks concurrently — one per core — so a 5-core executor runs 5 tasks simultaneously.

Two separate memory regions inside every executor 

┌─────────────────────────────────────────────────────────────┐
│                     EXECUTOR PROCESS                        │
│                                                             │
│  ┌───────────────────────┐   ┌───────────────────────────┐  │
│  │      JVM HEAP         │   │    OFF-HEAP (native)      │  │
│  │  spark.executor       │   │    spark.memory           │  │
│  │  .memory              │   │    .offHeap.size          │  │
│  │                       │   │                           │  │
│  │  • Execution memory   │   │  • Parquet column         │  │
│  │    (shuffles, joins,  │   │    buffers                │  │
│  │    sort, aggregation) │   │  • Arrow columnar         │  │
│  │  • Storage memory     │   │    encoding / decoding    │  │
│  │    (cached datasets)  │   │  • Decompression          │  │
│  │  • User code objects  │   │    scratch space          │  │
│  │                       │   │                           │  │
│  │  Managed by GC ✓      │   │  No GC — direct OS ✓     │  │
│  └───────────────────────┘   └───────────────────────────┘  │
│                                                             │
│  + JVM overhead (thread stacks, NIO buffers, JVM internals) │
└─────────────────────────────────────────────────────────────┘

The JVM heap is where Java objects live — DataFrames being processed, shuffle buffers, cached datasets. The JVM’s garbage collector (GC) automatically frees unused objects. GC is a blessing (no manual memory management) and a burden: when it runs, it pauses all threads in the executor. On a heap loaded with large byte arrays — like a 3 GB Parquet column chunk — GC pauses can last seconds, stalling tasks and triggering YARN timeouts.

Off-heap memory is allocated directly from the OS, bypassing the JVM entirely. The GC never sees it, never touches it, never pauses for it.

The silent failure mode: setting off-heap to 0 while Spark’s vectorized reader still tries to allocate native buffers can cause container-kill OOM crashes that produce no Java stack trace — the OS kills the process before the JVM can catch the error. The symptom is an executor that disappears from the YARN UI with no exception logged.
WorkloadOff-heap recommendation
Reading / writing Parquet or ORC2–8 GB depending on column width and file sizes
Heavy joins, window functionsErr toward more — large Arrow buffers during wide aggregations
Pure RDD operations (no structured API)0 — off-heap is never used
In-memory DataFrame ops, no file I/O0 or 1–2 GB as a safety margin
CSV → Parquet → Snowflake ETL (this project)4 GB — starting point for 2 TB/day at current scale

Executor Math

The Executor Math card shows the step-by-step results of the memory breakdown for a single executor slot and the key configuration values written to spark-defaults.conf. Every value updates on each slider change.

Metric Tiles

  • Executors/VM: floor((vCPUs − 2) ÷ exec_cores) — how many executors fit given the core budget. The −2 reserves vCPUs for the guest OS and YARN NodeManager daemon.
  • Total executors: (executors/VM × total VMs) − 1 — the −1 reserves one executor for the YARN cluster-mode application driver. This is the value written to spark.executor.instances.
  • Total executor cores: TOTAL_EXEC × exec_cores — feeds spark.default.parallelism = total_exec_cores × 2.
  • Slot size (GB): (RAM_VM − 2) ÷ EXEC_PER_VM — the RAM envelope available to one executor, including all overhead components.
  • executor.memory: the heap value written to spark-defaults.conf. This is what YARN sees as the memory request when launching the container.
  • OOM risk: LOW (green) / MARGINAL (amber) / HIGH (red). The threshold is skew-profile-dependent: 4 GB for Uniform, 5 GB for Moderate skew, 6 GB for Heavy skew (set in Section D). The active margin, threshold, and skew profile name are shown in the tile sub-label. Choosing a higher skew profile tightens the requirement — a 5 GB margin that would be green under Uniform shows as amber under Moderate skew.

Executor Memory Layout

The Executor Memory Layout card provides two complementary visualisations of how memory is structured within a worker VM.

Worker VM executor layout (upper section): a tile for each executor slot within the VM, plus a NodeManager tile. Each executor tile shows its core count and slot size in GB, colour-coded green (OOM margin adequate at the active threshold), amber (marginal), or red (OOM risk). The NodeManager tile accounts for the 2 vCPU / 2 GB overhead deducted before executors are sized.

Memory anatomy bar (lower section): a proportional horizontal bar showing how one slot’s total GB is divided into four segments:

  • Executor heap (blue) — spark.executor.memory: the JVM heap Spark requests from YARN.
  • JVM overhead (amber) — overhead_fraction × executor.memory: reserved for thread stacks, NIO buffers, and native JVM structures.
  • Off-heap (purple) — spark.memory.offHeap.size: vectorized Parquet read buffers outside the GC-managed heap. Absent when off-heap is set to 0.
  • OOM margin (green / amber / red matching the OOM risk tile) — the free space between the sum of the above segments and the slot ceiling.

Use this card to verify that the executor heap dominates the slot (large blue segment) and that the OOM margin is clearly visible. A very thin green segment is a warning sign even before the OOM risk tile turns amber.

Executor Core Sensitivity ±1

The Core Sensitivity card shows a three-column comparison of how key metrics change if you use one fewer or one more core per executor than the current setting. It answers: what do I actually gain or lose by moving the exec_cores slider one notch?

The three columns are:

  • N −1 — one fewer core per executor than current
  • N ★ Current — your current exec_cores setting (highlighted column)
  • N +1 — one more core per executor than current

Each row is colour-coded:

  • ● Safe — value is within healthy operating range
  • ● Marginal — approaching a threshold
  • ● Unsafe — value violates a threshold (OOM risk, overcommit, etc.)
  • ● Informational — metric has no threshold; shown for reference only

Use this card to evaluate trade-offs before committing to a slider change: fewer cores per executor gives more executors (more task parallelism, smaller heaps, lower GC risk); more cores gives fewer executors with larger memory slots (better for wide joins and large aggregations).

7. Section D — Storage & Workload Sizing

Section D brings together the OSD configuration that drives storage capacity planning and the workload-specific parameters that influence OOM thresholds, scratch adequacy, Parquet partitioning, and network transfer estimates. The left column contains the Storage & Ceph Configuration card; the right column contains the Workload Sizing card.

Storage & Ceph Configuration

The storage card provides a visual map of how drives are allocated on each node. The OSD count and replication factor controls in this card are the same inputs as in Section A — changing either updates both cards simultaneously.

  • 3× SSD (480 GB each): two drives form the Proxmox OS ZFS mirror; the third is a ZFS hot spare. All three are unavailable to Ceph or Spark.
  • NVMe OSDs (amber): drives assigned to Ceph. Each contributes 3.84 TB to the raw pool across all 3 nodes. Total raw = OSD_count × 3.84 TB × 3 nodes.
  • NVMe scratch (green): remaining drives (7 − OSD_count) divided across worker VMs. SCRATCH_TB_PER_VM = (7 − OSD_count) ÷ VMs_per_node × 3.84 TB. Used as spark.local.dir for shuffle spill.

Three metric tiles below the drive map show: Ceph usable TB (raw ÷ replication factor), days of ingest at the current daily volume, and scratch TB per VM.

Workload Sizing

The Workload Sizing card contains all the sliders that characterise the ETL job’s data. These inputs drive Ceph retention calculations, scratch adequacy checks, Parquet partition sizing, shuffle partition estimates, and the Network Path transfer time. Two selectors at the bottom of the slider group adjust OOM and scratch thresholds based on expected data behaviour.

Slider Range What it controls & what it affects
Daily data volume (TB) 0.5–4 TB, step 0.5 Total data ingested per day across all batches/dumps. Drives: Ceph days-of-ingest calculation, scratch adequacy threshold (daily_TB × shuffle_amp × join_factor), and the shuffle partition count estimate. Note: distinct from CSV dataset size (below), which represents one batch and drives Parquet partitioning math only.
CSV dataset size 1–10,000 GB, step 1 GB The total size of one CSV batch or dump. Drives Parquet partitioning math: INPUT_PARTS = ceil(CSV_MB ÷ chunk_MB) and PARQ_MB = CSV_MB ÷ compression. Equal to daily volume only when there is one dump per day. Default 3 GB reflects the current development workload.
Shuffle amplification 1–4×, step 0.5 How much data expands during shuffle operations (groupBy, join, sort). A simple filter amplifies ~1×. A join between two equal-sized tables approaches 2×. Multi-stage pipelines with skewed keys can reach 3–4×. Drives the scratch adequacy threshold and the shuffle partition count estimate via SHUFFLE_DATA_TB = daily_TB × amp.
Shuffle output size (MB) 64–512 MB, step 32 Maps to spark.sql.adaptive.advisoryPartitionSizeInBytes. After joins and aggregations produce many small shuffle partitions, AQE coalesces them toward this target size. This is the output-side / post-shuffle control only — it has no effect on how CSV files are read. CALC_PARTITIONS = ceil(SHUFFLE_DATA_MB ÷ target_MB).
Parquet compression 2×–10×, step 1 How much smaller the Parquet output is vs the CSV input, due to columnar encoding and Snappy compression. Typical business data compresses 4×–6×. Highly repetitive data can reach 8×–10×. PARQ_MB = CSV_MB ÷ ratio. Affects Parquet partition count and Network Path transfer time.
Read chunk size 64–256 MB, step 32 Maps to spark.sql.files.maxPartitionBytes. Spark splits the CSV batch into chunks so multiple executors can read in parallel. One read task is spawned per chunk. INPUT_PARTS = ceil(CSV_MB ÷ chunk_MB). Smaller = more tasks = more read parallelism; larger = fewer tasks = less scheduling overhead. Input-side control only — independent of Shuffle output size.

Data Skew Profile

The Data skew profile selector raises the OOM safety threshold used across the Diagnostics card, Executor Math OOM risk tile, and Core Sensitivity table.

ProfileOOM thresholdWhen to use
Uniform 4 GB Data is evenly distributed across keys. Executor heap pressure is predictable and consistent across tasks.
Moderate skew 5 GB Some keys have significantly higher row counts than others. Skewed partitions can momentarily spike heap usage above the average slot load.
Heavy skew 6 GB Severe key imbalance (e.g., null key aggregation, power-law distributions). Individual tasks may use several multiples of average memory, making OOM crashes likely even with comfortable average margins.

Selecting a higher skew profile does not change any computed memory values — it changes the threshold against which the OOM margin is compared. A 5 GB margin that shows green under Uniform shows amber under Moderate skew.

Join Type

The Join type selector applies a multiplier to the shuffle volume used in the scratch adequacy check and Operating Envelope. It does not change the computed shuffle volume — it tightens the threshold to account for the additional scratch pressure that sort-merge and many-to-many joins produce beyond the shuffle amplification estimate.

Join typeScratch factorWhen to use
Broadcast-heavy ×1.00 Small dimension tables fit in memory and are broadcast; the large fact table is never shuffled. Scratch volume closely matches the raw shuffle amplification estimate.
Mixed ×1.10 A combination of broadcast and sort-merge joins. Moderate additional scratch pressure from the sort-merge paths.
Wide shuffle joins ×1.25 Large fact-to-fact joins or many-to-many relationships that cannot be broadcast. Both sides are sorted and merged, writing both halves to scratch. Scratch pressure significantly exceeds the naive shuffle volume estimate.

The scratch adequacy check uses: SCRATCH_TB_PER_VM ≥ daily_TB × shuffle_amp × join_factor × 0.75. If the check fails, the join factor is shown in the Diagnostics warning message.

Output Metrics

Four tiles below the sliders summarise the Parquet partitioning math for the current CSV dataset size:

  • Parquet size (cluster): PARQ_MB = CSV_MB ÷ compression — the data volume that travels from Ceph RGW to Snowflake. Used in Section E Network Path transfer time.
  • Input partitions: ceil(CSV_MB ÷ max_part_MB) — how many parallel Spark tasks are spawned to read the CSV. Should be at least equal to total executor cores for full read parallelism.
  • Output partitions: ceil(PARQ_MB ÷ 150 MB) — the recommended df.repartition(N) value producing ~150 MB Parquet files. Shown in Section F Spark Configuration as a code-level recommendation.
  • Output file size: colour-coded. Green if 64–300 MB (Snowflake sweet spot). Amber if outside that range.
How Read chunk size and Shuffle output size relate: These two sliders look similar but govern completely different stages of a Spark job.

Read chunk size (spark.sql.files.maxPartitionBytes) is the input-side control. Spark splits your CSV batch into chunks of at most this size and spawns one read task per chunk. It has no effect on what happens after the shuffle.

Shuffle output size (spark.sql.adaptive.advisoryPartitionSizeInBytes) is the output-side / post-shuffle control. After joins and aggregations produce many small shuffle partitions, AQE coalesces them toward this target size. It has no effect on the read phase. 

CSV files → [Read chunk size] → read tasks → shuffle → [Shuffle output size via AQE] → output tasks

Both default to 128 MB, which is why they appear identical — but they solve different problems at different points in job execution. Tune them independently.
Snowflake COPY INTO file size guidance: Snowflake performs best with files between 100–256 MB. Files below 64 MB create excessive metadata overhead and increase the number of micro-partitions Snowflake must manage. Files above 300 MB reduce Snowflake’s ability to parallelize the load across warehouse threads. The output file size tile turns amber when outside this range.

8. Section E — Network Path

This card models the three-segment network path from Ceph RGW to Snowflake on Azure. The bottleneck is always the slowest segment, and all timing calculations use 80% of bottleneck capacity as the effective throughput to model realistic sustained utilization under TCP overhead and competing traffic.

Important: Ceph RGW must have a publicly reachable S3 endpoint for Snowflake to execute COPY INTO directly. Snowflake pulls the data from your S3-compatible endpoint — it does not receive a push from the Spark job. The ETL pattern is: Spark writes Parquet to Ceph RGW → job completes → Snowflake task runs COPY INTO my_table FROM @my_ceph_stage → Snowflake reads the files over the public internet path modelled here.

Slider Range What it controls & what it affects
Node → Switch (Gbps) 10–100 Gbps, step 10 Per-node NIC speed northbound to the top-of-rack switch. The calculator aggregates all 3 nodes: AGG = speed × 3. So 10 Gbps/node = 30 Gbps aggregate. If the aggregate node bandwidth exceeds the switch uplink capacity, the switch becomes the bottleneck even before traffic reaches the WAN.
Switch → Core (Gbps) 10–100 Gbps, step 10 The switch uplink to the core/border router. If lower than the aggregate node capacity, outbound traffic is queued and throttled at this segment. This is a common bottleneck in on-premises environments where ToR switches have 40 Gbps uplinks serving 4× 10 Gbps server ports.
Core → Snowflake (Mbps) 100–10,000 Mbps, step 100 The WAN or internet circuit speed to Azure where Snowflake is hosted. Displayed in Mbps for precision at sub-1 Gbps values. Converted internally: NET_WAN_G = Mbps ÷ 1000. This segment is almost always the bottleneck for cloud-destined traffic in on-premises deployments.

Path Diagram

Each segment in the diagram shows its effective speed (aggregate bandwidth for the Node→Switch segment). The bottleneck segment is highlighted in red; non-bottleneck segments are highlighted in green, making it immediately obvious which infrastructure investment would have the greatest impact on transfer times.

Output Metrics

  • Bottleneck: identifies which of the three segments is the constraining factor.
  • Eff. throughput: BOTTLENECK_G × 0.80 — realistic sustained rate at 80% utilization, accounting for TCP handshaking, retransmission overhead, and background traffic.
  • Transfer time: PARQ_GB ÷ (EFF_GBPS ÷ 8) — time to move one Parquet load from Ceph RGW to Snowflake. Uses the Parquet size from Section D Workload Sizing (not the raw CSV size).
  • Re-run window: transfer_time × 2 — worst-case time budget if a job fails and must be re-run. Used in SLA planning: if the re-run window exceeds the acceptable ETL window, consider increasing WAN bandwidth or reducing daily data volume.
Default workload sizing: At the default 1,000 Mbps WAN link with a 3 GB CSV batch (approximately 600 MB Parquet at 5× compression), the transfer takes only a few seconds at 80% utilization. The 1 Gbps link is fully adequate for this workload. The bottleneck would shift to the WAN only if daily CSV input exceeded ~40 GB at 5× compression.

10. How Everything Connects — The Calculation Chain

Every slider feeds into a deterministic calculation chain. Understanding the chain helps you predict the effect of any change before making it. Here is a worked example tracing what happens when you increase spark.executor.cores from 4 (current default) to 5:

Example: Increasing exec_cores from 4 to 5

1. EXEC_PER_VM = floor((vCPUs − 2) ÷ 5) // fewer executors per VM than at ÷4
2. TOTAL_EXEC = (EXEC_PER_VM × total_VMs) − 1 // fewer total executors cluster-wide
3. SLOT_GB = (RAM_VM − 2) ÷ EXEC_PER_VM // slot grows because fewer executors share VM RAM
4. EXEC_MEM = floor(((SLOT_GB − offheap) ÷ (1 + overhead)) × 0.95) // larger heap per executor
5. OOM_MARGIN = SLOT_GB − (EXEC_MEM × (1 + overhead) + offheap) // margin widens → OOM risk improves
6. DEFAULT_PAR = TOTAL_EXEC × 5 × 2 // spark.default.parallelism changes
7. SHUFFLE_PARTS recalculates from daily_TB and target_partition_size // independent of exec_cores
8. Warnings re-evaluate: low executor density warning may appear if EXEC_PER_VM drops below 3

The net effect: each executor gets more cores and more memory, making it better suited to large joins and wide aggregations, but there are fewer executors total so less task-level parallelism. The current default of exec_cores = 4 is well-suited to the dev ETL workload (CSV → Parquet). Increase to 5 if you observe task-level bottlenecks from insufficient core parallelism; increase further only if you see significant GC overhead from very large heaps.

The RAM/VM Pre-Scale Chain

There is one additional cascade that runs before the main calculation: the RAM/VM pre-scale block. It fires whenever any slider changes and must run first because it modifies another slider’s range:

  1. User moves the RAM/node slider (Section A) from 256 GB to 320 GB.
  2. The pre-scale block runs immediately: it computes the new RAM/VM max = floor((320 − reservations) ÷ VMs_per_node / 32) × 32.
  3. The current RAM/VM value is proportionally rescaled: new_val = round(current_pct × new_max / 32) × 32, keeping the slider at the same relative position even though the range has changed.
  4. All downstream executor math (SLOT_GB, EXEC_MEM, OOM_MARGIN, everything in Section C) recalculates with the new RAM/VM value.
  5. The Warnings card re-evaluates all checks. If Section C is collapsed, its UPDATED badge fires.

This ordering is why state restoration (loading a saved scenario) must restore ram_node before iterating through other range inputs — if ram_per_vm is restored before ram_node sets the correct ceiling, the RAM/VM slider may clamp to the wrong maximum and the restored configuration will differ from the saved one.

Healthy configuration checklist: A well-tuned configuration shows all green in Diagnostics & Warnings: no core or RAM overcommit — OOM margin above the active skew-profile threshold (4 / 5 / 6 GB for Uniform / Moderate / Heavy) — Ceph retention ≥ 7 days at current daily ingest volume — scratch TB per VM ≥ 75% of adjusted shuffle volume (daily_TB × shuffle_amp × join_factor) — at least 3 executors per VM for task-level isolation — AQE enabled for adaptive partition coalescing — NUMA pinning enabled (even on single-socket dev hardware, for consistency with production).
