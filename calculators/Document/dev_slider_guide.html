1. The Mental Model

Every node in the development cluster holds a fixed amount of hardware: cores, RAM, and NVMe drives. Before Apache Spark can use any of it, several infrastructure services must take their share first — the Proxmox hypervisor, the Ceph storage daemons, and the YARN resource manager. What remains after those reservations is what Spark actually gets to work with.

The sliders control how those remaining resources are divided. Think of it as a series of nested decisions, each one made inside the constraints set by the previous one:

The yellow items are fixed by the hardware and software configuration — they cannot be changed through the calculator. The blue items are sliders. The green items are outputs — what results from your choices.

Key insight: Because the layers are nested, a change at an outer layer ripples inward. Adding more Ceph OSDs (Layer 2) reduces the cores and RAM available for Spark VMs (Layer 3), which reduces how many executors fit (Layer 4) — all automatically, all immediately.

2. What Is Baked Into the Hardware

Several values in the calculator are not sliders — they are physical facts about the machines that cannot be changed through software. These are shown in the fixed hardware strip at the top of the calculator and are built into every calculation.

Fixed valueDev clusterWhy it matters
Nodes 3 Total executor count and Ceph replication both assume exactly 3 nodes. Ceph requires a minimum of 3 for 3× replication.
Cores per node 32 Single AMD EPYC socket. Hard ceiling on all core budgets. After infrastructure reservations, roughly 24 cores remain for VMs.
NUMA domains per node 1 Single socket means all 32 cores share one memory bus. No NUMA-crossing penalty, but also no ability to pin a VM to a sub-socket domain. Max vCPUs per VM is 32.
NVMe drives per node 7 × 3.84 TB All NVMe is shared between Ceph OSDs and Spark scratch. The OSD count slider decides the split. Scratch drives run JBOD/XFS — no hardware RAID. Each scratch NVMe is an independent spark.local.dir path. These drives are not interchangeable with the SSDs.
SSD drives per node 3 × 480 GB Dedicated to Proxmox OS (ZFS mirror, 2 drives) and hot spare (1 drive). Never available to Ceph or Spark. Not a tunable resource.
Infrastructure reservations Proxmox 2c/8 GB · RGW 4c/8 GB · MON 2c/6 GB · ZFS ARC 0c/8 GB Fixed software allocations required to run the platform. These are consumed before any Spark VM is considered. ZFS ARC (the OS-mirror read cache) is capped at 8 GB via zfs_arc_max and reserved explicitly. The OSD reservation (1c/3 GB per drive) scales with the OSD count slider.
The RAM slider is a partial exception. RAM/node is adjustable because DIMMs can be added or removed. However the step size is fixed at 32 GB (one DIMM), and the ceiling is set by the physical DIMM slots available (384 GB for the dev cluster at full population).

3. Group 1 — Node Memory Budget

This slider sets the total RAM available on each node. Everything else — infrastructure reservations, VM allocations, executor sizing — is calculated as a fraction of this value.

GROUP 1 NODE MEMORY BUDGET Sets the overall pool before anything is divided
RAM / Node 128–384 GB · step 32 GB default 384 GB
How much physical RAM each node contributes to the pool. One step equals one 32 GB DIMM. The calculator subtracts fixed infrastructure reservations from this value first, then divides what remains across VMs. At 384 GB (full population) the cluster operates at peak memory bandwidth — fewer DIMMs means less bandwidth as well as less capacity.
▼ Drives: RAM/VM ceiling (auto-adjusts) · total RAM available after reservations · executor slot size

4. Group 2 — Storage Split

The 7 NVMe drives per node serve two completely different purposes: Ceph OSDs (the distributed object store that provides the S3 endpoint) and Spark scratch (local temporary storage for shuffle data during jobs). These two uses compete directly — every drive you give to Ceph is one fewer drive available for shuffle spill, and vice versa. The sliders in this group control that trade-off. Scratch drives are configured as JBOD (no hardware RAID) and formatted XFS.

Why Spark needs local scratch: When a Spark job performs a join or group-by across a large dataset, it must physically move data between executors (this is called a shuffle). That intermediate data is written to local disk before being sent across the network. If the local scratch space runs out mid-shuffle, the job fails.
GROUP 2 STORAGE SPLIT NVMe drives: Ceph OSDs vs Spark scratch
NVMe drives → Ceph OSDs 0–7 · step 1 default 5
How many of the 7 NVMe drives per node are dedicated to Ceph storage. Each OSD adds 3.84 TB to the Ceph raw pool (shared across 3 nodes, divided by the replication factor to get usable capacity). Each OSD also consumes 1 core and 3 GB RAM from the node budget — those resources are no longer available to Spark VMs. The remaining drives (7 − OSD count) are divided across worker VMs as scratch space.
▼ Drives: Ceph usable capacity · scratch TB per VM · core and RAM budgets (via OSD reservation) · RAM/VM ceiling
Ceph replication factor 2–3 · step 1 default 3
How many copies of every object Ceph maintains across the 3 nodes. At 3× (default), each byte of data is written to all 3 nodes — the cluster survives one full node failure with no data loss. At 2×, usable capacity increases by 50% but one node failure leaves data at risk. Replication factor directly divides the raw OSD capacity to produce usable storage.

Reducing below 3 on a 3-node cluster is generally inadvisable unless capacity is critically constrained.
▼ Drives: Ceph usable TB · days of retention warning threshold

5. Group 3 — VM Shape

Proxmox runs Spark inside virtual machines (VMs). These sliders control how the resources left over after infrastructure reservations are carved into VMs. The shape of the VM determines the maximum size of the YARN containers (executors) that can run inside it.

Why VMs and not bare metal? Proxmox uses KVM virtualisation to isolate workloads, manage resources, and enable live migration. Spark itself runs inside the VM via YARN, which treats the VM’s vCPUs and RAM as its total available hardware — it has no visibility of the physical node underneath.
GROUP 3 VM SHAPE How remaining node resources are carved into worker VMs
VMs per node 1–2 · step 1 default 1
How many Spark worker VMs share each physical node. With 1 VM, a single VM receives all available cores and RAM after infrastructure reservations — maximum executor density per node. With 2 VMs, resources are split equally, which allows workload isolation or running two different Spark versions simultaneously, but each VM gets half the cores, half the RAM, and half the scratch NVMe.
▼ Drives: vCPUs/VM ceiling · RAM/VM ceiling · scratch NVMe per VM · total executor count
vCPUs per VM 8–32 · step 2 default 14
How many virtual CPU cores the VM presents to YARN. YARN treats this as its total core budget. The slider ceiling is 32 — the single-socket hardware boundary — but the practical safe maximum is lower and changes with your OSD count.

Infrastructure services consume cores before the VM gets any. With the default of 5 OSDs: Proxmox (2) + Ceph RGW (4) + Ceph MON (2) + 5 OSDs (5) = 13 cores reserved, leaving a practical ceiling of 19 vCPUs. Add or remove OSDs and that ceiling shifts by 1 core per OSD.

Of the vCPUs assigned to the VM, a further 2 are reserved for the guest OS and YARN NodeManager daemon, so only vCPUs − 2 are available to Spark executors.

Auto-clamp behaviour: unlike the old implementation, this slider does auto-clamp when the practical ceiling drops. Every time recalc() runs it recomputes the available core ceiling and forces the slider to that value if the current setting exceeds it. In practice this means increasing the OSD count immediately drops the vCPUs/VM slider to the new ceiling — no manual correction needed and no silent overcommit. The Fixed Overhead Reservations card still shows the current ceiling for reference.

The one case that does not self-correct: a saved scenario loaded from localStorage may contain a vCPU value that was valid when it was saved but is now over-ceiling (for example, if the hardware configuration changed between sessions). On load, recalc() will clamp the value silently — the restored slider position will be lower than what was saved. If a scenario loads and the vCPU count looks lower than expected, compare it against the Fixed Overhead Reservations card to confirm.
▼ Drives: Executors per VM (via exec_cores) · total executor count · core budget bar
RAM per VM 64–dynamic max · step 32 GB default 320 GB
How much RAM the VM is allocated. This slider’s maximum is computed dynamically — it automatically adjusts whenever you change RAM/node, OSD count, VMs/node, or the services VM toggle. You cannot accidentally set it above what the hardware supports. Of the RAM assigned to the VM, 2 GB is reserved for the YARN NodeManager, and the remainder is divided into executor slots.

Reference figures with 4 OSDs and full 384 GB RAM per node: fixed reservations total approximately 44 GB (Proxmox 8 GB + ZFS ARC 8 GB + Ceph RGW 8 GB + Ceph MON 6 GB + 4 OSDs×3 GB + ResourceManager ~2 GB), leaving approximately 340 GB available for VMs. Compare to the old model (before ZFS ARC was added) which showed ~36 GB reserved and ~348 GB available.
▲ Constrained by: RAM/node · OSD count · VMs/node · services VM toggle
▼ Drives: Executor slot size (GB) · executor heap · OOM margin
NUMA pinning on / off default on
On the dev cluster (single socket), NUMA pinning has no practical effect on memory locality — there is only one NUMA domain. It is enabled by default for consistency with the production cluster, where pinning a VM to a single socket prevents expensive cross-socket memory accesses. Disabling it on dev is harmless but generates an advisory notice.
▼ Drives: Warning state only (dev cluster)
AQE (Adaptive Query Execution) on / off default on
When ON, Spark dynamically coalesces small shuffle partitions at runtime based on actual data sizes, rather than using a fixed partition count. This is critical for ETL workloads where data volume varies between runs. When OFF, the calculated partition count is used as a fixed value — often too many for small datasets, too few for large ones. Disabling triggers a yellow warning.
▼ Drives: spark.sql.shuffle.partitions behaviour · warning state

6. Group 4 — Executor Shape

YARN divides the VM’s resources into executor containers — the actual JVM processes that run Spark tasks. These sliders control how large each executor is and how its memory is internally divided. This is where the most consequential tuning decisions happen.

Why executor shape matters more than count: More executors means more parallelism but smaller heaps and more GC pressure. Fewer executors means larger heaps (better for wide joins) but less concurrency. There is no universally correct answer — it depends on the workload.
GROUP 4 EXECUTOR SHAPE How the VM is sliced into Spark executor containers
Executor cores 2–10 · step 1 default 4
How many vCPUs each executor uses. This directly determines how many executors fit in the VM: floor((vCPUs − 2) ÷ exec_cores). Fewer cores per executor = more executors = more task parallelism but smaller memory per executor. More cores = fewer executors = larger memory slots, better for wide joins. 4 is the current default — a good balance for this cluster’s RAM-to-core ratio.
▲ Constrained by: vCPUs per VM
▼ Drives: Executors per VM · total executor count · executor slot size · all memory calculations
YARN overhead % 8–20% · step 1% default 10%
YARN adds a memory overhead on top of every executor container to cover JVM process memory, internal buffers, and OS overhead. This is configured via spark.executor.memoryOverheadFactor. The overhead is calculated as a percentage of executor heap size. Setting it too low can cause YARN to kill containers that exceed their allocation; too high wastes RAM that could otherwise go to heap.
▼ Drives: Effective container size · OOM margin · max executor memory
Off-heap memory 0–16 GB · step 2 GB default 6 GB
Memory allocated outside the JVM heap for Spark’s vectorized Parquet reader and Apache Arrow columnar buffers. This memory is not managed by the Java garbage collector — it is allocated directly from the OS and freed immediately when no longer needed. Every GB here reduces the heap available to spark.executor.memory. Setting to 0 disables off-heap entirely, which can cause silent OOM crashes when reading Parquet files (the OS kills the process before the JVM can log an error).
▼ Drives: Executor heap size · OOM margin · spark.memory.offHeap.size config value

7. Group 5 — Workload Sizing

These sliders do not change how the hardware is configured — they tell the calculator how hard you plan to hit it. They drive the scratch adequacy check, the Ceph capacity warning, and the partition count recommendations. Getting these right means the warnings are meaningful; leaving them at wrong values means warnings fire (or don’t fire) incorrectly.

GROUP 5 WORKLOAD SIZING Describes the data and job characteristics — not the hardware
Daily ingest volume 0.5–4 TB · step 0.5 TB default 2 TB
How much raw data arrives each day. This drives the Ceph days-of-retention calculation and sets the baseline for the shuffle scratch warning. If you are running smaller development batches, reduce this to get accurate scratch and retention warnings for your actual workload size.

Retention formula (current): the calculator does not simply divide usable Ceph capacity by daily volume. It first subtracts two additional reservations: VM OS disks (VMS_NODE × NODES × 80 GB) stored on Ceph RBD, and Parquet outputs pending outbound transfer (N_PARQUET × (DAILY_TB ÷ PARQ_COMP)). Only the remaining CEPH_EFFECTIVE capacity is divided by daily volume to produce CEPH_DAYS. This means increasing daily volume tightens retention faster than you might expect, because it simultaneously grows both the denominator and the Parquet-pending reservation in the numerator.
▼ Drives: Ceph retention warning · scratch adequacy warning · shuffle data volume · partition count
Shuffle amplification 1–4× · step 0.5 default 3.5×
How much data expands during shuffle operations (joins, group-bys, sorts). A simple filter is ~1×. A join between two equal-sized tables approaches 2×. Multi-stage pipelines with sort-merge joins and aggregations can reach 3–4×. The calculator uses this to estimate worst-case scratch space required: daily_TB × shuffle_amp. If your actual jobs spill more than your scratch can hold, the job fails. The default of 3.5× reflects the development ETL pipeline’s mix of join types.

Compression note: Spark’s default shuffle compression (Snappy) is always active. On-disk shuffle data typically occupies 40–60% of the calculated ceiling, not the full value. The displayed scratch requirement is therefore a conservative planning figure — the actual on-disk usage will be materially lower. The sub-text beneath the scratch adequacy bar reads “compression active: effective ceiling ~2× higher” to reflect this. Do not reduce the amplification slider to compensate; keep it at the logical data expansion rate and let the compression note explain the headroom.
▼ Drives: Scratch adequacy warning · CALC_PARTITIONS · spark.sql.shuffle.partitions
Data skew profile Uniform / Moderate / Heavy default Uniform
How evenly your data is distributed across partition keys. Uniform: tasks finish in similar times — no straggler problem. Moderate: some keys are more common but manageable with AQE. Heavy: a small number of keys dominate, causing some executors to process 10× more data than others — the job’s runtime is determined by the slowest task. Heavy skew raises the OOM warning threshold because the overloaded executor needs more headroom.
▼ Drives: OOM warning threshold (4 / 5 / 6 GB depending on skew) · warning state
Join type Broadcast / Mixed / Wide shuffle default Broadcast
The dominant join strategy your jobs use. Broadcast joins send the smaller table to every executor in full — no shuffle required, very fast, but the small table must fit in executor memory. Mixed: a combination of broadcast and sort-merge joins. Wide shuffle joins: both tables are large, requiring a full cross-network shuffle before joining. Wide shuffle joins generate significantly more scratch data and network traffic.
▼ Drives: Scratch warning multiplier · network estimator

8. Group 6 — Partition Strategy

These sliders control how Spark divides its work into parallel tasks — both when reading input data and when writing shuffle output. They do not affect hardware allocation but they determine whether Spark under- or over-parallelises the job, and they feed directly into the generated spark-defaults.conf values.

GROUP 6 PARTITION STRATEGY Read parallelism and shuffle output sizing
Shuffle output size 64–512 MB · step 32 MB default 192 MB
Maps to spark.sql.adaptive.advisoryPartitionSizeInBytes. After a join or aggregation, AQE coalesces small shuffle output partitions back toward this target size. This is an output-side / post-shuffle control — it has no effect on how input files are read. Larger values produce fewer, bigger output partitions (less scheduling overhead). Smaller values produce more, smaller partitions (more parallelism at the cost of overhead).
▼ Drives: spark.sql.adaptive.advisoryPartitionSizeInBytes · output file count estimate
CSV dataset size 1–10,000 GB · step 1 GB default 3 GB
The total size of the raw CSV batch being read each run. Used in the Parquet Partitioning card to model the CSV → Parquet transformation and estimate the resulting Parquet file size, output partition count, and Snowflake COPY INTO transfer time. Does not affect executor sizing or scratch calculations — those use the Daily ingest slider instead.
▼ Drives: Parquet output size estimate · Snowflake transfer time · input partition count
Parquet compression ratio 2–10× · step 1 default 5×
How much smaller the Parquet output is compared to the CSV input, due to columnar encoding and Snappy compression. Typical business data compresses 4–6×. Highly repetitive data (low-cardinality categoricals, many NULLs) can reach 8–10×. Drives the Parquet size estimate, transfer time calculation, and the Parquet-pending Ceph reservation (PARQ_PENDING_TB = N_PARQUET × (DAILY_TB ÷ PARQ_COMP)).
▼ Drives: Parquet output MB estimate · Snowflake transfer time · PARQ_PENDING_TB (Ceph reservation)
Parquet outputs pending transfer 1–5 · step 1 default 2
How many complete Parquet output sets may accumulate on Ceph at the same time before a prior set is confirmed transferred to Snowflake. Each set represents one day’s worth of compressed output: DAILY_TB ÷ PARQ_COMP. The total Ceph capacity reserved for pending outputs is N_PARQUET × (DAILY_TB ÷ PARQ_COMP) (PARQ_PENDING_TB). This is subtracted from usable Ceph capacity before computing CEPH_DAYS.

Set this to the maximum number of Parquet batches you expect to be resident on Ceph simultaneously — for example, 2 if your outbound transfer pipeline confirms one batch before the next ETL run completes, or 3–5 if transfers may lag behind by multiple days (slow Snowflake COPY INTO, network issues, or a manual review step). Underestimating this value produces an optimistic retention figure; overestimating it reserves capacity unnecessarily.
▲ Constrained by: Daily ingest volume · Parquet compression ratio (both determine per-set size)
▼ Drives: PARQ_PENDING_TB · CEPH_EFFECTIVE · CEPH_DAYS · Ceph retention warning
Read chunk size 64–256 MB · step 32 MB default 128 MB
Maps to spark.sql.files.maxPartitionBytes. When Spark reads your CSV batch it cannot hand the entire file to one executor — it splits it into chunks so multiple executors can work in parallel. This slider sets the upper bound on each chunk’s size. One read task is spawned per chunk. Example: a 2,500 MB CSV at 128 MB gives 20 parallel read tasks. This is an input-side / read-phase control only — it has no effect on shuffle output or post-join partition sizing (that is Shuffle output size). Smaller values = more tasks = more parallelism at the cost of scheduling overhead.
▼ Drives: spark.sql.files.maxPartitionBytes · input task count

Section F — Spark Configuration Output

The calculator’s Section F is not a slider group — it is a read-only output panel generated by buildParamTable(). It produces a live spark-defaults.conf-style table that updates automatically whenever any slider changes.

What it shows: The table lists the derived Spark configuration parameters — including executor memory, cores, overhead, off-heap size, partition counts, and shuffle tuning parameters — formatted as key = value pairs suitable for pasting directly into spark-defaults.conf or a Spark submit command. It is a copy-paste reference, not an interactive input.

Key parameters included in the output:

  • spark.executor.memory and spark.executor.cores — derived from the executor shape sliders
  • spark.executor.memoryOverheadFactor — from the YARN overhead % slider
  • spark.memory.offHeap.size — from the off-heap GB slider
  • spark.sql.shuffle.partitions — calculated from total executor count × cores
  • spark.sql.adaptive.advisoryPartitionSizeInBytes — from the shuffle output size slider
  • Shuffle tuning parameters: serializer, lz4.blockSize, shuffle.file.buffer, memoryFraction, maxSizeInFlight
  • spark.sql.parquet.compression.codec set to snappy for Snowflake compatibility

Because all values are recomputed on every slider change, Section F always reflects the current slider state. Use it to review the full parameter set before applying it to the cluster.

9. Slider Dependencies at a Glance

Moving one slider can silently change what another slider is allowed to be, or change an output without any visible warning. The table below captures the most important of these relationships.

RAM / Node RAM/VM ceiling (auto-clamps) The VM cannot be allocated more RAM than the node has after reservations. The slider physically moves.
OSD count Core & RAM budgets · RAM/VM ceiling · scratch TB/VM Each OSD consumes 1 core + 3 GB RAM. More OSDs = less left for VMs. Also shifts NVMe drives away from scratch.
VMs per node RAM/VM ceiling · vCPUs/VM ceiling · scratch/VM Resources are divided equally across VMs. Two VMs each get half.
vCPUs per VM Executors per VM · total executor count EXEC_PER_VM = floor((vCPUs − 2) ÷ exec_cores). Fewer vCPUs = fewer executor slots.
RAM per VM Slot size · executor heap · OOM margin Slot size = RAM_VM − 2 GB (NodeManager) ÷ executors per VM. All memory metrics flow from this.

10. The Three Things That Go Wrong

Almost every warning in the calculator traces back to one of three underlying problems. Each warning is the end result of a chain of slider decisions — understanding the chain tells you which slider to move, and in which direction, to resolve it.

⚠ Out-of-Memory (OOM) Crash
What you see: The OOM margin row in the Executor Math card turns red. In a live job, an executor disappears from the YARN UI without a logged error, or the job fails with a container-killed message.

What is happening: The memory slot allocated to each executor is too small to hold the data it is processing, plus the JVM’s own overhead, plus the off-heap buffers. YARN enforces strict container memory limits — when an executor exceeds its allocation, YARN kills it immediately.

The cause chain:
  • OSD count ↑ → cores and RAM reserved for Ceph ↑ → RAM available for VMs ↓ → RAM/VM ceiling ↓ → slot size ↓ → OOM margin ↓
  • Executor cores ↑ → fewer executors per VM → each executor gets a larger slot → OOM margin improves
  • Executor cores ↓ → more executors per VM → each slot is smaller → OOM margin tightens
  • Off-heap ↑ → off-heap carved from slot before heap → less heap per executor → OOM margin tightens
  • Overhead % ↑ → more YARN container overhead → less headroom → OOM margin tightens
The silent variant: Setting off-heap to 0 while running Parquet workloads can cause the OS to kill the executor process before the JVM logs anything. The symptom is an executor that simply vanishes from the YARN UI with no stack trace. This is not an OOM warning — it is a silent crash that is easy to misdiagnose.

Warning threshold: OOM margin < 4 GB (uniform skew) · < 5 GB (moderate skew) · < 6 GB (heavy skew). The skew profile slider raises the threshold because skewed tasks process more data per executor than the average.
Move to fix: RAM/VM ↑ · Executor cores ↑ (fewer, larger slots) · Off-heap ↓ · Overhead % ↓ · OSD count ↓ (frees RAM budget)
⚠ Scratch Space Exhaustion
What you see: The scratch warning fires in the Diagnostics card. In a live job, a shuffle stage fails with a disk-full or “no space left on device” error. Unlike OOM, there is no retry — the job aborts.

What is happening: During shuffle operations (joins, group-bys, sorts), Spark writes intermediate data to the local NVMe scratch drives before sending it across the network. If that intermediate data exceeds the available scratch space, the write fails and the job dies.

The cause chain:
  • OSD count ↑ → more NVMe drives go to Ceph → fewer drives left for scratch → scratch TB/VM ↓
  • Daily ingest ↑ → more data to shuffle → worst-case scratch requirement ↑
  • Shuffle amp ↑ → each byte of input produces more shuffle data → worst-case scratch requirement ↑
  • VMs per node ↑ → scratch NVMe split across more VMs → scratch TB/VM ↓
  • Join type → wide shuffle joins multiply scratch requirement beyond the basic shuffle_amp estimate
Warning threshold: Scratch TB per VM < 75% of worst-case shuffle volume (daily_TB × shuffle_amp). The 75% buffer exists because shuffle data is not perfectly distributed — some VMs will receive more than the average.

Compression context: The displayed scratch requirement is calculated from the logical data volume before shuffle compression. Spark’s default Snappy shuffle compression means actual on-disk usage is typically 40–60% lower than the calculated figure — the scratch adequacy bar sub-text notes “compression active: effective ceiling ~2× higher.” In practice this warning rarely becomes a real-world disk-full event when compression is active, but the conservative figure is retained as a planning guarantee: if on-disk usage somehow approaches the calculated ceiling (e.g. compression-resistant data, disabled codec), the cluster still has headroom.
Move to fix: OSD count ↓ (frees NVMe for scratch) · VMs per node ↓ · Daily ingest and Shuffle amp ↑ to model your real workload accurately
⚠ Insufficient Ceph Retention
What you see: The Ceph retention warning fires in the Diagnostics card. In a live cluster, this means the S3 object store is filling up faster than data is being archived or deleted — eventually new ingest jobs cannot write and begin failing.

What is happening: Ceph’s usable capacity is fixed by how many NVMe drives you assign to it and the replication factor. If daily ingest consumes capacity faster than you’re clearing old data, the store fills up.

The cause chain:
  • OSD count ↓ → fewer NVMe drives in Ceph → raw Ceph capacity ↓ → usable capacity ↓ → fewer effective days
  • Replication factor ↑ → each byte written N times → usable capacity = raw ÷ N ↓
  • Daily ingest ↑ → more data consumed per day and Parquet-pending reservation grows → effective headroom ↓ faster than the raw numbers suggest
  • N_PARQUET ↑ → more Parquet sets held on Ceph awaiting transfer confirmation → PARQ_PENDING_TB ↑ → CEPH_EFFECTIVE ↓
  • VMs/node ↑ → more VM OS disks on Ceph RBD → VM_OS_USABLE_TB ↑ → CEPH_EFFECTIVE ↓
Current formula: CEPH_EFFECTIVE = CEPH_USABLE − VM_OS_USABLE_TB − PARQ_PENDING_TB; CEPH_DAYS = CEPH_EFFECTIVE ÷ DAILY_TB. The old formula was simply CEPH_USABLE ÷ DAILY_TB — it did not account for Ceph capacity consumed by infrastructure.

Warning threshold: CEPH_EFFECTIVE ÷ DAILY_TB < 3 days. Three days is the minimum buffer to survive a weekend without active data management.
Move to fix: OSD count ↑ (more Ceph capacity) · Replication factor ↓ to 2× (with caution on a 3-node cluster) · Daily ingest ↓ to reflect actual load
⚠ Core Budget Overcommit
What you see: The core budget bar turns red. This is the most direct cause-and-effect warning in the calculator: you have allocated more vCPUs to the VM than the node has available after infrastructure services take their share.

What is happening: Physical cores are shared between Proxmox, Ceph daemons, and the Spark VM. If the VM is promised more vCPUs than physically exist after infrastructure reservations, the host CPU is overcommitted. Spark tasks and Ceph OSD processes compete for the same physical cores, causing latency spikes, slow shuffles, and unpredictable job runtimes.

The cause chain:
  • OSD count ↑ → each OSD consumes 1 physical core → cores available for VM ↓ → vCPUs/VM ceiling ↓
  • vCPUs/VM slider auto-clamps on every recalc() call → increasing OSD count immediately drops the slider to the new ceiling, so live overcommit cannot persist through the UI
Concrete example: At 5 OSDs (default), infrastructure consumes 13 cores, leaving 19 for the VM. Setting vCPUs to 20 causes the bar to turn red briefly — then on the next recalc() the slider drops to 19 and the bar recovers. The transient red flash is the only visible indication.

The one remaining failure mode: a scenario saved in localStorage when the core ceiling was higher (e.g. fewer OSDs at save time) will restore the old vCPU value, which recalc() silently clamps downward on load. The restored configuration is valid, but the vCPU count will be lower than what was saved — check the Fixed Overhead Reservations card if a loaded scenario looks different than expected.

Resolution: Check the Fixed Overhead Reservations card for the current available core count. The slider ceiling is enforced automatically; no manual correction is needed for live use.
Move to fix: vCPUs/VM ↓ to stay within (32 − infrastructure cores) · OSD count ↓ (each reduction frees 1 core)
The fundamental tension: OSD count is the slider that touches all four warnings simultaneously. More OSDs → better Ceph retention, worse scratch, less RAM for executors, fewer available cores for the VM. Less OSDs → more scratch, more executor RAM, more VM cores, but reduced Ceph capacity. The calculator shows all warnings live so you can find the OSD count that satisfies all constraints at once — that is its primary purpose.
