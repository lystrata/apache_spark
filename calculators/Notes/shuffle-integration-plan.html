A Scope & Objectives · What this plan covers · What it does not change

This plan integrates the findings from the 2026-04-19 shuffle amplification research (8 sources, 2 rounds) into the project's calculators and documentation. Objectives:

  • Create a Developer Guide on shuffle optimization practices specific to our hardware and scratch constraint
  • Create a Research Summary for IT admin / vendor distribution
  • Update existing documents with new findings, compression notes, and parameter corrections
  • Add footnotes with works cited where research findings are asserted
  • Add compression context and updated parameter output to all three calculators

What does not change: The core scratch formula (2× factor is validated). No hardware recommendations change. No calculator slider ranges change. This is a documentation and annotation pass, plus targeted parameter updates in the spark-defaults.conf output sections.

Notes/spark-shuffle-developer-guide.html
CREATE

Audience: ETL developers writing jobs on the cluster.

Sections:

  • Why Shuffle Matters Here — the 4-drive RAID-5 scratch constraint, what the ceiling means at 2 TB/day vs. 5 TB/day
  • What You Can Control — developer practices that reduce amplification:
    • reduceByKey over groupByKey (with amplification impact) [4][6]
    • Broadcast hints for small dimension tables, threshold guidance [4][5][7]
    • Filter / predicate pushdown before joins [4]
    • Repartition on join key before multi-join pipelines [7]
    • Bucketing for hot join paths — eliminate shuffle entirely [4]
    • Avoiding unnecessary collect(), count(), distinct() [4]
  • What the Cluster Does For You — AQE auto-broadcast, skew splitting, partition coalescing [5]
  • Reading the Spark UI — Shuffle Read/Write, Spill (Disk) columns; identifying skew [4][7]
  • Configuration Reference — recommended spark-defaults.conf parameters with explanations [2][5][6]
  • Works Cited
